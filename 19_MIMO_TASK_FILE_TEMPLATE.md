# 19 — Шаблон task-файла для MiMo

Этот файл нужен, чтобы другой ИИ-агент сразу понял, как правильно создавать задания для MiMo в проекте DVRK Control.

Главное правило: пользователь не должен вручную разбираться в коде. ИИ должен подготовить один понятный task-файл `vXXX.md`, пользователь копирует команду в SSH, MiMo читает файл и выполняет задачу автономно.

---

## 1. Правильная схема работы

Работа ведётся так:

```text
Пользователь формулирует задачу
↓
ИИ уточняет только действительно важные детали
↓
ИИ создаёт task-файл /opt/dvrk_control_app/vXXX.md
↓
Пользователь запускает MiMo
↓
MiMo выполняет задачу на VPS
↓
MiMo создаёт report_vXXX.md
↓
ИИ проверяет report, screenshots, логику и только потом принимает задачу
```

---

## 2. Что не нужно делать

Не нужно:

```text
давать пользователю длинные инструкции по ручному редактированию кода;
просить пользователя открывать много файлов;
давать сразу 5–10 задач;
прыгать далеко вперёд;
писать расплывчатую задачу без проверок;
принимать PASS MiMo без проверки.
```

---

## 3. Формат ответа пользователю

Когда нужно дать задачу для MiMo, правильный формат ответа:

```text
Коротко: что делаем и зачем.

Скопируй в SSH:

```bash
cd /opt/dvrk_control_app

cat > /opt/dvrk_control_app/vXXX.md <<'EOF'
...
EOF
```

Потом в MiMo напиши:

Прочитай vXXX.md и выполни автономно.
```

Но если пользователь просит “создай файл”, нужно создавать файл напрямую, а не печатать весь текст в чат.

```

```

---

## 5. Нумерация задач

Использовать следующий свободный номер:

```text
v169.md
v170.md
v171.md
...
```

Не использовать номер, если он уже занят.

Если задача не запускалась, не считать её выполненной.

Пример:

```text
v166 formula reconciliation был предложен, но не запускался.
```

---

## 6. Обязательная структура task-файла

Каждый task-файл должен содержать разделы:

```text
# vXXX — Название задачи

## 1. Цель
## 2. Контекст
## 3. Что нельзя ломать
## 4. Backup
## 5. Что изменить
## 6. Что не трогать
## 7. Backend-проверки
## 8. Browser / DOM проверки
## 9. Screenshots
## 10. Report
## 11. Git commit / tag
```

---

## 7. Раздел “Цель”

Цель должна быть конкретной.

Плохо:

```text
улучшить сайт
```

Хорошо:

```text
Расширить страницу /integrations: добавить полноценные блоки YooKassa и Tilda, не трогая /logs и не вызывая внешние API на GET.
```

---

## 8. Раздел “Контекст”

В контексте нужно объяснить, почему задача делается.

Пример:

```text
В v168 была создана страница /integrations, но она пока слишком простая.
Нужно расширить её до полноценной панели подключений YooKassa и Tilda.
```

---

## 9. Раздел “Что нельзя ломать”

Этот раздел обязателен.

Пример:

```text
Нельзя:
- показывать secrets полностью;
- затирать secret пустым input;
- вызывать YooKassa/Tilda API на GET /integrations;
- ломать /orders, /chz, /stats, /customers, /wb, /logs;
- менять логику закрывающего чека;
- менять ZH-правило Честного Знака.
```

---

## 10. Раздел “Backup”

Для рискованных задач писать:

```text
Перед изменениями сделать backup.
Зафиксировать путь backup в report_vXXX.md.
```

Особенно обязательно для:

```text
миграций БД;
YooKassa;
Tilda webhook;
WB нормализации;
backup-системы;
массовых пересчётов;
удаления данных.
```

---

## 11. Раздел “Что изменить”

Писать конкретно:

```text
Добавить поля:
- API URL
- callback URL
- last receipt
- last error

Добавить кнопки:
- Сохранить
- Проверить подключение
- Скопировать webhook URL
```

Не писать абстрактно.

---

## 12. Раздел “Что не трогать”

Всегда указывать соседние зоны, которые нельзя менять.

Пример:

```text
Не трогать:
- WB formulas;
- closing receipt payload;
- Tilda webhook parser;
- /logs layout;
- user_ui_settings v167.
```

---

## 13. Backend-проверки

Типовые проверки:

```text
python -m py_compile ...
smoke-test;
проверка миграций;
проверка GET no external API;
проверка secret masking;
проверка empty secret does not overwrite;
проверка audit без secrets.
```

---

## 14. Browser / DOM проверки

Для UI задач обязательно:

```text
открыть страницу в браузере;
проверить DOM;
проверить отсутствие JS errors;
проверить видимость основных блоков;
проверить кнопки;
проверить таблицы;
сделать screenshots.
```

Для Playwright указывать конкретные страницы.

---

## 15. Screenshots

В task-файле всегда требовать screenshots в `/tmp`.

Пример:

```text
/tmp/v169_integrations_main.png
/tmp/v169_integrations_yookassa.png
/tmp/v169_integrations_tilda.png
/tmp/v169_integrations_audit.png
```

Без screenshots UI-задачу не принимать.

---

## 16. Report

MiMo должен создать:

```text
/opt/dvrk_control_app/report_vXXX.md
```

В report обязательно:

```text
что изменено;
какие файлы изменены;
какие проверки прошли;
какие browser screenshots;
что не трогалось;
секреты не видны;
GET no external API;
известные ограничения;
git commit;
git tag.
```

---

## 17. Git

После принятой задачи нужен commit и tag.

Пример:

```bash
git status
git add ...
git commit -m "v169: expand integrations panel"
git tag v1.66-integrations-expanded
```

Tag должен быть осмысленным.

---

## 18. Шаблон vXXX.md

Ниже универсальный шаблон.

```markdown
# vXXX — Короткое название задачи

## 1. Цель

Коротко и конкретно описать, что нужно сделать.

## 2. Контекст

Объяснить текущую ситуацию и почему задача нужна.

## 3. Что нельзя ломать

Нельзя:
- ...
- ...
- ...

## 4. Backup

Перед изменениями:
- сделать backup, если задача рискованная;
- указать путь backup в report_vXXX.md.

## 5. Что изменить

Сделать:
1. ...
2. ...
3. ...

## 6. Что не трогать

Не менять:
- ...
- ...
- ...

## 7. Backend-проверки

Проверить:
- py_compile;
- smoke;
- миграции;
- GET no external API;
- security checks.

## 8. Browser / DOM проверки

Проверить через браузер:
- страница открывается;
- нет JS errors;
- основные блоки видны;
- таблицы не сломаны;
- buttons работают;
- secrets не видны.

## 9. Screenshots

Сохранить:
- /tmp/vXXX_page_1.png
- /tmp/vXXX_page_2.png

## 10. Report

Создать:
- /opt/dvrk_control_app/report_vXXX.md

В report указать:
- что изменено;
- какие проверки прошли;
- screenshots;
- known limitations;
- git commit;
- git tag.

## 11. Git

После успешных проверок:
- сделать commit;
- создать tag.
```

---

## 19. Пример будущей задачи v169

Логичный следующий task:

```text
v169 — расширить /integrations
```

Короткая цель:

```text
Расширить страницу /integrations до полноценной панели YooKassa/Tilda: добавить API URL, webhook/callback URL, last receipt, last error, critical Tilda settings, copy buttons и audit, сохранив secret masking и GET no external API.
```

---

## 20. Главное резюме

Кратко:

```text
Одна задача = один vXXX.md.
MiMo выполняет автономно.
UI задачи требуют screenshots.
PASS без проверки не принимать.
GET-страницы не вызывают внешние API.
Secrets не показывать и не логировать.
После задачи нужен report, commit и tag.
```
