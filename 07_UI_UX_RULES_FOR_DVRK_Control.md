# 07 — UI / UX правила для DVRK Control

Этот файл описывает правила интерфейса DVRK Control: как должны выглядеть страницы, какие элементы нужны пользователю, как оформлять таблицы, подсказки, статусы, кнопки и предупреждения.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в шаблонах, CSS, JS, страницах `/orders`, `/chz`, `/stats`, `/customers`, `/logs`, `/integrations`, `/wb` и других UI-разделах.

---

## 1. Главный принцип интерфейса

DVRK Control — это рабочий инструмент оператора.

Интерфейс должен быть:

```text
понятным;
компактным;
русским;
тёмным;
аккуратным;
без лишнего визуального шума;
без необходимости догадываться, что делает кнопка.
```

Пользователь не хочет интерфейс, где нужно долго разбираться. Но он также не хочет, чтобы каждая страница была забита огромными поясняющими блоками.

---

## 2. Язык интерфейса

Основной язык интерфейса:

```text
Русский
```

Английские технические слова допустимы только там, где они являются названием API, поля, статуса или технического термина:

```text
forPay
bankPaymentSum
full_prepayment
full_payment
webhook
raw_json
rrdId
```

Но рядом желательно давать понятное русское объяснение.

Пример:

```text
WB начислил forPay / 1
```

А не просто:

```text
forPay / item
```

---

## 3. Стиль интерфейса

Основной стиль:

```text
тёмная тема;
чёрный / графитовый фон;
тонкие границы;
сдержанные акценты;
синие активные элементы;
зелёные success-статусы;
жёлтые warning;
красные error.
```

Страницы должны выглядеть как единая система, а не как набор случайных шаблонов.

---

## 4. Центровка страниц

Пользователь предпочитает:

```text
страницы визуально центрированы;
контент не прилипает к краю экрана;
ширина контейнера контролируемая;
крупные таблицы могут растягиваться, но не должны ломать layout.
```

Если страница широкая, нужно использовать аккуратный wrapper:

```text
max-width;
margin auto;
адекватные отступы;
горизонтальный scroll только там, где он нужен.
```

---

## 5. Скроллбары

Пользователь не любит стандартные белые скроллбары.

Нужно использовать:

```text
тёмные;
тонкие;
спокойные;
в общей гамме сайта.
```

Особенно это важно для:

```text
таблиц;
логов;
backup console;
raw/debug блоков;
длинных карточек.
```

---

## 6. Подсказки

Пользователь хочет, чтобы рядом с неочевидными кнопками, статусами и блоками были пояснения.

Но важное уточнение:

```text
Не засорять интерфейс постоянными поясняющими блоками.
```

Лучший формат:

```text
tooltip при наведении;
короткая подпись;
иконка "i";
маленькое inline-пояснение.
```

Для неочевидных действий:

```text
должна быть подсказка, что произойдёт после нажатия.
```

Пример:

```text
"Пометить решённой"
tooltip: "Скрывает проблему из активного списка, но не удаляет запись из аудита."
```

---

## 7. Когда нужен постоянный видимый блок

Постоянный информационный блок нужен только если информация критична.

Например:

```text
YooKassa работает в тестовом режиме.
Реальные чеки не отправляются.

Tilda должна отправлять webhook только после оплаты.

Report-level расходы WB не распределены по товарам.
```

Но такие блоки должны быть компактными.

---

## 8. Кнопки

Кнопки должны быть понятными.

Не делать кнопки с неоднозначным текстом.

Плохо:

```text
OK
Run
Do
Apply
```

Лучше:

```text
Сохранить настройки
Проверить подключение
Отправить закрывающий чек
Сбросить настройки страницы
Скачать backup
Открыть логи интеграций
```

Опасные действия должны иметь confirm.

Примеры опасных действий:

```text
отправить чек;
сбросить настройки;
удалить заказ;
перезаписать secret;
повторить отправку после ошибки.
```

---

## 9. Цвета кнопок

Рекомендуемая логика:

```text
primary / blue   — основное безопасное действие;
green            — success / подтверждено;
yellow / amber   — предупреждение / требует внимания;
red              — опасное действие;
gray             — вторичное действие.
```

Не использовать красный для обычных кнопок.

---

## 10. Disabled-состояния

Если действие нельзя выполнить, кнопка должна быть disabled.

Но рядом нужно объяснить почему.

Пример:

```text
[Отправить закрывающий чек] disabled
Причина: "Введите код ЧЗ для товара 1007ZH"
```

Или через tooltip:

```text
Кнопка недоступна: сумма чека не совпадает с оплатой.
```

---

## 11. Статусы

Статусы должны быть человекочитаемые.

Плохо:

```text
status=2
unknown
pending
```

Лучше:

```text
Нужен код ЧЗ
Готов к закрывающему чеку
Закрывающий чек отправлен
Чек не нужен
Проверить признак расчёта
YooKassa pending
YooKassa succeeded
```

Технический статус можно оставить как вторичный текст.

---

## 12. Таблицы

Таблицы — основной формат работы в системе.

Правила:

```text
компактные строки;
липкие/понятные заголовки, если таблица длинная;
не раздувать высоту;
длинный текст обрезать;
полный текст показывать через tooltip/expand;
действия держать в правой колонке;
важные суммы выравнивать по правому краю;
статусы оформлять badge/pill.
```

Не делать огромные полотна.

Для `/logs` уже было исправлено:

```text
события слева;
backup и помощник справа;
таблица компактная;
page size 50/100/200.
```

---

## 13. Не ломать HTML-таблицы

Для обычных таблиц не использовать CSS grid на `table`, `tr`, `td`, если это ломает поведение.

Особенно осторожно с:

```text
/chz таблицы;
/orders таблицы;
/wb таблицы;
таблицы с действиями и статусами.
```

---

## 14. Страницы с большим количеством информации

Если информации много, использовать:

```text
карточки;
tabs;
details/collapse;
две колонки;
правый sidebar;
компактные summary cards.
```

Не складывать всё вертикально в один длинный поток.

Пример правильного направления:

```text
/logs после v168:
верхние summary cards;
слева события;
справа backup, быстрые действия, помощник.
```

---

## 15. Dashboard-страницы

Dashboard-страницы должны иметь:

```text
заголовок;
короткий подзаголовок;
верхние summary cards;
основной рабочий блок;
быстрые действия;
предупреждения только если нужны.
```

Для таких страниц важно сразу видеть главное, не скроллить вниз.

---

## 16. Страница /stats

Статистика должна быть информативной, но не хаотичной.

Важно:

```text
метрики должны быть сгруппированы;
скидки показывать отдельно;
товары до скидки / после скидки не смешивать;
доставка отдельно;
выручка оплачено отдельно;
настройки периода сохраняются через v167;
раскрытые/свёрнутые блоки могут сохраняться.
```

Не делать ложных предупреждений по каталогу.

---

## 17. Страница /customers

CRM / покупатели должна быть аккуратной.

Важно:

```text
не определять пол по имени;
не делать маркетинговые выводы без согласия;
не смешивать покупателей по имени;
email приоритетнее phone для key;
поиск/фильтры сохраняются через v167;
персональные данные не логировать лишний раз.
```

---

## 18. Страница /orders

Заказы — рабочая очередь.

Важно показывать:

```text
Tilda ID;
покупатель;
телефон;
email;
адрес;
доставка;
статус;
ЧЗ;
товары;
сумма;
дата;
действия.
```

Покупатель отображается так:

```text
delivery_fio
↓
customer_name
↓
пусто
```

Оператор должен быстро понимать, что делать с заказом.

---

## 19. Страница /chz

Страница ЧЗ должна показывать, какие заказы требуют действия.

Статусы должны быть понятные:

```text
Нужен код ЧЗ
Готов к закрывающему чеку
Закрывающий чек без ЧЗ
Чек не нужен
Проверить признак расчёта
```

Не показывать просто технические флаги без объяснения.

---

## 20. Карточка заказа /orders/{id}

Карточка заказа должна быть структурирована:

```text
Покупатель
Доставка
Оплата
YooKassa
Товары / позиции
Быстрые действия
Аудит / события
```

Финансы должны показывать:

```text
товары до скидки;
скидка / промокод;
товары после скидки;
доставка;
итого оплачено.
```

---

## 21. Карточка ЧЗ /chz/orders/{id}

Это критичная рабочая страница.

Должно быть понятно:

```text
нужен ли чек;
нужен ли ЧЗ;
какой товар требует ЧЗ;
куда вставить код;
какая сумма уйдёт в чек;
какой payment_id YooKassa;
почему кнопка disabled, если нельзя отправить.
```

Перед отправкой нужен confirm.

---

## 22. Страница /logs

После v168 цель страницы:

```text
не длинное полотно;
компактный dashboard;
события слева;
backup и системный помощник справа;
фильтры в одну строку;
таблица событий с page size.
```

Не возвращать длинное вертикальное полотно.

---

## 23. Страница /integrations

Страница `/integrations` создана для настроек подключения:

```text
YooKassa;
Tilda;
webhook;
общие настройки;
история изменений.
```

Требования UI:

```text
секреты замаскированы;
настройки сгруппированы;
есть кнопки сохранить/проверить;
есть критические настройки Tilda;
GET не вызывает внешние API.
```

---

## 24. WB-страницы

WB-модуль отдельный.

Важно:

```text
не смешивать WB с Tilda/YooKassa/ЧЗ;
не вызывать WB API на GET;
показывать forPay понятным русским текстом;
расходы WB показывать со знаком минус;
report-level расходы не размазывать без отдельного режима.
```

На таблицах WB пользователь хочет видеть:

```text
− Комиссия WB / 1
− Эквайринг / 1
+/- Корректировка до forPay / 1
WB начислил forPay / 1
− Логистика WB / 1
− Обратная логистика WB / 1
Остаток после WB / 1
```

---

## 25. Знаки денег

Деньги должны отображаться логично.

Для расходов:

```text
− Комиссия WB / 1
− Эквайринг / 1
− Логистика WB / 1
− Обратная логистика WB / 1
− Удержания
− Штрафы
```

Для начислений:

```text
WB начислил forPay / 1
+ Доплаты
+ SPP / корректировка до forPay, если она объясняет forPay
```

Важно:

```text
raw значения в БД можно хранить как есть;
UI должен показывать смысл для продавца.
```

---

## 26. Tooltips вместо лишних блоков

Для большинства пояснений использовать tooltip.

Примеры, где tooltip нужен:

```text
WB начислил forPay / 1
Корректировка до forPay / 1
Остаток после WB / 1
Невыкуп / обратная логистика qty
Проверить признак расчёта
Чек не нужен
```

Постоянный блок нужен только для критичной информации.

---

## 27. Неочевидные последствия действий

Если кнопка меняет состояние, обязательно объяснить последствия.

Примеры:

```text
"Сбросить настройки страницы"
Подсказка/confirm:
Будут сброшены только настройки интерфейса этой страницы. Данные, заказы, чеки и покупатели не удаляются.

"Отправить закрывающий чек"
Confirm:
Будет отправлен закрывающий чек через YooKassa.

"Удалить заказ"
Confirm:
Заказ будет скрыт/удалён из рабочей очереди. Проверьте, что это тестовый заказ.
```

---

## 28. Сохранение UI-настроек

В v167 сделано сохранение UI-настроек только для:

```text
/stats
/customers
```

Сохраняется:

```text
/stats:
  period;
  custom period;
  collapsed sections;
  section order;
  metric cards.

/customers:
  search;
  period;
  segment;
  section;
  marketing filter.
```

Важно:

```text
не подключать сохранение UI-настроек ко всему сайту без отдельного решения;
URL params имеют приоритет над сохранёнными настройками;
токены и secrets нельзя сохранять как UI-настройки.
```

---

## 29. Мобильность

Основной сценарий — desktop browser.

Но страницы не должны полностью ломаться на меньшей ширине.

Минимально:

```text
контейнеры не вылезают;
таблицы получают горизонтальный scroll;
кнопки не перекрывают текст;
карточки складываются в одну колонку.
```

---

## 30. Проверки UI

После любых изменений UI нужно делать browser/DOM проверки.

Желательно через Playwright.

Проверять:

```text
страница открывается;
нет JS errors;
основные блоки видны;
кнопки на месте;
секреты не видны;
таблица не ломается;
GET не вызывает внешние API;
скриншоты сохранены в /tmp.
```

Для задач MiMo нужно требовать screenshots:

```text
/tmp/vXXX_page_name.png
```

---

## 31. Что писать в report_vXXX.md

Если задача меняет UI, отчёт должен содержать:

```text
что изменено;
какие страницы затронуты;
какие страницы не затронуты;
какие browser-проверки прошли;
где screenshots;
были ли JS errors;
не появились ли внешние API вызовы на GET;
не видны ли secrets;
git commit и tag.
```

---

## 32. Главное резюме

Кратко:

```text
Интерфейс DVRK Control должен быть тёмным, русским, компактным и понятным.
Не делать длинные полотна.
Сложную информацию группировать.
Неочевидное объяснять через tooltip.
Опасные действия подтверждать confirm.
Расходы показывать со знаком минус.
Секреты не показывать.
GET-страницы не вызывают внешние API.
UI изменения обязательно проверять browser/DOM тестами.
```
