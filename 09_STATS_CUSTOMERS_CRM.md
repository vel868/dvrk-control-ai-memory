# 09 — Статистика и Покупатели / CRM

Этот файл описывает разделы `/stats` и `/customers` в проекте DVRK Control: какие данные они показывают, как считаются финансы, как устроена CRM-логика, как обрабатываются скидки, доставки, секции товаров, согласия покупателей и сохранение UI-настроек после v167.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в статистике, CRM, покупателях, финансовых метриках, скидках, промокодах, секциях товаров, согласиях, фильтрах и сохранении настроек интерфейса.

---

## 1. Роль `/stats`

Страница `/stats` — это основная управленческая статистика по заказам Tilda / DVRK Control.

Она должна отвечать на вопросы:

```text
сколько заказов;
сколько оплачено;
какие товары продаются;
какие секции продаются;
какие скидки применялись;
какая доставка;
какие товары требуют ЧЗ;
какие товары CLUB;
какие размеры / секции;
какие месяцы и периоды лучше;
какие промокоды использовались.
```

Это не WB-аналитика. WB вынесен в отдельный модуль `/wb`.

---

## 2. Роль `/customers`

Страница `/customers` — это CRM / Покупатели.

Она должна отвечать на вопросы:

```text
кто покупал;
сколько заказов у покупателя;
сколько он оплатил;
какой средний чек;
какие товары покупал;
какие размеры / секции;
есть ли CLUB;
нужен ли ЧЗ в его заказах;
какие согласия есть;
можно ли использовать маркетинг.
```

CRM строится из заказов Tilda / DVRK Control.

---

## 3. Что нельзя смешивать

Нельзя смешивать:

```text
заказы Tilda / DVRK Control
и
WB-продажи
```

WB-модуль отдельный:

```text
/wb
/wb/products
/wb/unit-economics
/wb/warehouses
/wb/product-warehouses
```

Статистика `/stats` и CRM `/customers` работают по заказам сайта / Tilda, а не по WB-финансовым отчётам.

---

## 4. Финансовая модель заказов

В статистике и CRM используется стабильная модель:

```text
goods_gross_sum      — товары до скидки
goods_discount_sum   — скидка / промокод
goods_net_sum        — товары после скидки
delivery_sum         — доставка
order_total_paid     — фактически оплачено
```

Главная формула:

```text
goods_net_sum + delivery_sum = order_total_paid
```

Эта формула должна соблюдаться в:

```text
/stats
/customers
/orders/{id}
/chz/orders/{id}
закрывающем чеке YooKassa
```

---

## 5. Скидки и промокоды

Скидки нельзя прятать.

В интерфейсе должно быть понятно:

```text
Товары до скидки
Скидка / промокод
Товары после скидки
Доставка
Итого оплачено
```

Скидка распределяется только между товарами.

Доставка не участвует в скидке.

Формула распределения:

```text
item_discount = goods_discount_sum * item_gross / goods_gross_sum
item_net_sum = item_gross - item_discount
```

Если товар один, вся скидка уходит на него.

---

## 6. Пример скидки

Пример заказа:

```text
Товар: 299.00
Промокод: 29.90
Товар после скидки: 269.10
Доставка: 1053.03
Итого оплачено: 1322.13
```

В статистике и CRM должно быть видно:

```text
товарная выручка до скидки;
размер скидки;
товарная выручка после скидки;
доставка отдельно;
общая оплаченная сумма.
```

## 7. Что показывать на `/stats`

На `/stats` должны быть блоки по смыслу:

```text
сводные карточки;
продажи по товарам;
финансы;
бизнес-метрики;
ЧЗ;
география;
динамика;
месяцы;
ассортимент;

размеры;
товары / заказы по секциям;
CLUB и секции;
контроль каталога;
промокоды и скидки.
```

Не обязательно все блоки всегда раскрывать. Пользователь предпочитает компактный интерфейс.


## 9. Периоды статистики

На `/stats` важен выбранный период.

Период может быть:

```text
сегодня;
1 день;
2 дня;
3 дня;
7 дней;
30 дней;
свой период.
```

Если в URL явно указан период:

```text
/stats?period=30d
```

то URL имеет приоритет над сохранёнными настройками.

Если URL без параметров:

```text
/stats
```

то применяется сохранённая настройка v167.

---

## 10. Финансовые блоки `/stats`

Финансы должны показываться не одной суммой, а раздельно:

```text
Товары до скидок
Скидки / промокоды
Товары после скидок
Сумма доставки
Общая выручка оплачено
```

Не смешивать доставку с товарной выручкой.

Не считать скидку как расход доставки.

---

## 11. Товарная выручка

Товарная выручка для статистики должна использовать net после скидки.

То есть если товар продан со скидкой:

```text
в статистике товара должна учитываться сумма после распределения скидки.
```

Но для анализа скидок нужно также сохранять:

```text
gross;
discount;
net.
```

---

## 12. Секции товаров

SKU-правила:

```text
1xxx = Мужское
2xxx = Женское
5xxx = Аксессуары
ZH   = Честный Знак
CLUB = клуб / акция
```

Примеры:

```text
1007_CLUB     = Мужское + CLUB
1007ZH_CLUB   = Мужское + ЧЗ + CLUB
2008_CLUB     = Женское + CLUB
2008ZH_CLUB   = Женское + ЧЗ + CLUB
5010_CLUB     = Аксессуары + CLUB
5006ZH        = Аксессуары + ЧЗ
DELIVERY      = Доставка
```

---

## 13. Важно про секции

Пользователь сам решает, где товар находится в каталоге.

Нельзя считать ошибкой:

```text
кепка в аксессуарах;
жилетка в аксессуарах;
товар с размером в аксессуарах;
нестандартное название в секции.
```

Система не должна спорить с каталогом пользователя.

---

## 14. Контроль каталога

Блок "Контроль каталога" должен быть нейтральным.

Не использовать формулировки:

```text
Проблема
Рекомендация
```

Не ругаться на:

```text
Кепка в аксессуарах
Жилет в аксессуарах
Размер у аксессуара
```

Оставлять только объективные проверки:

```text
SKU без секции;
CLUB без базовой секции;
одежда без размера, если это действительно применимо;
доставка ошибочно как товар, если такое появилось.
```

---

## 15. CLUB-логика

`CLUB` — это акционная / клубная механика.

Она может быть у товара любой секции:

```text
Мужское + CLUB
Женское + CLUB
Аксессуары + CLUB
```

`CLUB` не означает ЧЗ.

ЧЗ определяется только по `ZH`.

---

## 16. ЧЗ в статистике

Статистика должна уметь показывать:

```text
сколько заказов с ЧЗ;
сколько товаров требует ЧЗ;
сколько закрывающих чеков отправлено;
сколько ожидает кода;
сколько чеков не нужно;
сколько unknown payment_mode.
```

Но не нужно смешивать это с WB-маркировкой.

---

## 17. `/customers` — принцип CRM

Покупатели агрегируются из заказов.

Главный ключ склейки:

```text
normalized email first
phone second
```

То есть:

```text
если есть email — использовать email;
если email нет — использовать телефон;
не склеивать только по имени.
```

Имя может совпадать у разных людей, поэтому имя не является надёжным ключом.

## 22. Мужское / Женское — это не пол покупателя

Очень важно:

```text
Мужское и Женское в системе — это секции товара.
Это не пол покупателя.
```

Нельзя определять пол покупателя по:

```text
имени;
покупаемой секции;
товарам;
размеру.
```

Не писать:

```text
покупатель мужчина;
покупатель женщина.
```

Если нужен маркетинговый сегмент, он должен быть нейтральным:

```text
покупал товары из секции Мужское;
покупал товары из секции Женское;
покупал аксессуары.
```

---

## 23. Маркетинг и согласия

Маркетинг разрешён только при наличии согласия.

В проекте учитываются согласия:

```text
Oferta_and_Soglashenia
Person_data
Marketing
```

Смысл:

```text
Oferta_and_Soglashenia — согласие с публичной офертой и пользовательским соглашением;
Person_data — согласие на обработку персональных данных;
Marketing — согласие на рассылки.
```

Маркетинг:

```text
можно использовать только если Marketing = true;
если Marketing нет — не предлагать рассылку;
система сама ничего не отправляет клиентам.
```

---

## 24. Что нельзя делать в CRM

Нельзя:

```text
определять пол по имени;
склеивать людей только по имени;
делать маркетинговую рассылку без согласия;
автоматически отправлять сообщения покупателям;
показывать лишние персональные данные в логах;
сохранять персональные данные как UI-настройки.
```

## 25. Поиск и фильтры `/customers`

На `/customers` могут быть:

```text
поиск;
период;
segment;
section;
marketing filter;
sort column;
sort direction;
page size.
```

После v167 сохраняются UI-настройки `/customers`.

Сохраняться могут:

```text
search;
period;
segment;
section;
marketing filter.
```

Строку поиска сохранять можно, потому что это UI-состояние оператора.

Но нельзя сохранять сами данные покупателя как настройку.

## 26. URL priority на `/customers`

Если пользователь открыл:

```text
/customers?search=abc
```

то URL имеет приоритет над сохранённым поиском.

Если открыл просто:

```text
/customers
```

то применяются сохранённые настройки v167.

## 27. UI-настройки v167

В v167 была создана таблица:

```text
user_ui_settings
```

Она используется только для:

```text
/stats
/customers
```

Не подключать её ко всему сайту без отдельной задачи.

Разрешённые pages:

```text
stats
customers
```

Другие pages должны отклоняться.

---

## 28. Что хранится в user_ui_settings

Поля таблицы:

```text
id
scope
page
setting_key
setting_value JSONB
updated_at
created_at
```

Уникальный индекс:

```text
(scope, page, setting_key)
```

`scope` пока используется как:

```text
default
```

В будущем можно заменить на user_id, если появится авторизация.

---

## 29. Что нельзя хранить в UI settings

Нельзя хранить:

```text
token;
secret;
password;
authorization;
cookie;
YooKassa credentials;
WB token;
Tilda secret;
полные персональные данные покупателей.
```

Для `/customers` допустима только строка поиска и фильтры как UI-состояние.

## 34. Связь с `/chz`

ЧЗ-статусы могут использоваться в статистике.

Если меняется ЧЗ-логика, проверить:


---

## 35. Связь с `/integrations`

Страница `/integrations` не должна ломать статистику и CRM.

Но там могут храниться настройки Tilda/YooKassa, которые косвенно влияют на новые заказы.

GET `/integrations` не должен вызывать внешние API и не должен менять статистику.
