# 1. Что такое DVRK Control

**DVRK Control** — это внутренняя web-система пользователя для управления заказами, Честным Знаком, YooKassa, Tilda, аналитикой, покупателями, логами, backup и Wildberries-аналитикой.

Проект работает как FastAPI-приложение на VPS.

Основная рабочая папка:

```text
/opt/dvrk_control_app
```

Dev-домен:

```text
https://control-dev.bussanmaster.ru
```

Systemd-сервис:

```text
dvrk-control-dev
```

Текущий важный tag на момент передачи:

```text
v1.65-logs-integrations-settings
```

# 2. Для чего нужна система

Система нужна, чтобы пользователь мог управлять интернет-магазином и маркетплейсами без ручного хаоса.

Главные задачи:

```text
1. Принимать оплаченные заказы из Tilda.
2. Понимать, каким товарам нужен Честный Знак.
3. Давать оператору рабочее место для ввода кодов ЧЗ.
4. Отправлять второй закрывающий чек через YooKassa.
5. Видеть заказы, статусы, покупателей и статистику.
6. Вести аудит действий и системные логи.
7. Делать и скачивать backup.
8. Анализировать Wildberries по финансовым отчётам.
9. Видеть экономику товаров WB: forPay, логистика, расходы, остаток после WB.
10. Постепенно прийти к чистой прибыли по каждому товару.
```

---

# 3. Главный бизнес-процесс Tilda / YooKassa / ЧЗ

Критичная стабильная схема:

```text
Tilda принимает оплату.
Tilda формирует первый чек предоплаты.
DVRK Control формирует второй / закрывающий чек через YooKassa.
```

DVRK Control **не должен делать первый чек**. Первый чек уже делает Tilda через YooKassa.

DVRK Control делает только закрывающий чек после того, как:

```text
1. заказ оплачен;
2. система поняла, нужен ли Честный Знак;
3. оператор ввёл код ЧЗ для маркированных товаров;
4. либо оператор подтвердил, что ЧЗ не нужен для обычных товаров.
```

Признак маркированного товара — суффикс `ZH` в SKU.

Примеры:

```text
5006ZH       = товар требует ЧЗ
5006         = товар не требует ЧЗ
1007ZH_CLUB  = товар требует ЧЗ и является CLUB-товаром
DELIVERY     = доставка, не товар
```

---

## `/chz` — Чеки и ЧЗ

Рабочий раздел для маркированных товаров и закрывающих чеков.

Система определяет состояние заказа:

```text
ZH + full_prepayment + нет кода ЧЗ → нужен код ЧЗ
ZH + все коды введены → можно отправлять закрывающий чек
full_prepayment без ZH → закрывающий чек без ЧЗ
full_payment → закрывающий чек не нужен
unknown payment mode → нужно проверить признак расчёта
```

---

## `/chz/orders/{id}` — Рабочее место закрывающего чека

Здесь оператор:

```text
видит состав заказа;
вводит код Честного Знака;
проверяет суммы;
подтверждает отправку второго чека;
видит ответ YooKassa.
```

Кнопка отправки закрывающего чека должна быть активна только когда:

```text
заказ требует закрывающий чек;
суммы сходятся;
для ZH-товаров введены коды ЧЗ;
оператор поставил подтверждающий checkbox.
```

# 5. Wildberries: общий смысл

Для финансов WB используются не оперативные заказы, а финансовые отчёты реализации.

Ключевой API:

```text
POST /api/finance/v1/sales-reports/detailed
```

Важные поля WB:

```text
retailAmount                — цена продажи покупателю
ppvzSalesCommission          — комиссия WB
acquiringFee                 — эквайринг
forPay                       — WB начислил продавцу по товарной строке
deliveryService              — логистика WB
rebillLogisticCost           — доначисление / перерасчёт логистики
paidStorage                  — хранение
paidAcceptance               — приёмка
deduction                    — удержания
penalty                      — штрафы
additionalPayment            — доплаты
bankPaymentSum               — сумма к оплате на уровне отчёта
```

Критично:

```text
deliveryAmount и returnAmount — это не рубли, а количества / признаки доставки и возврата.
```

---

# 6. Что такое forPay

`forPay` — это сумма, которую WB начислил продавцу по товарной продаже.

Упрощённая воронка:

```text
Цена продажи
− Комиссия WB
− Эквайринг
+/- Корректировка до forPay
= WB начислил forPay
```

Потом:

```text
WB начислил forPay
− Логистика WB
− Обратная логистика WB
− Корректировки логистики / rebill
− Хранение
− Приёмка
− Удержания товара
− Штрафы товара
+ Доплаты товара
= Остаток после WB
```

`forPay` — это **не чистая прибыль** и **не точный банковский перевод за штуку**.

Банковская выплата (`bankPaymentSum`) доступна на уровне отчёта, а не точно на уровне каждой штуки товара.

# 9. Как пользователь предпочитает работать

Пользователь не хочет получать хаотичные команды.

Правильный формат следующей разработки:

```text
1. Обсудить задачу.
2. Дать один bash-блок, который создаёт /opt/dvrk_control_app/vXXX.md.
3. В конце дать короткую команду для MiMo:
   Прочитай vXXX.md и выполни автономно.
4. Пользователь запускает MiMo.
5. MiMo создаёт report_vXXX.md.
6. ChatGPT проверяет report и скрины.
7. Только после этого задача принимается или отправляется на доработку.
```

```text

предлагать переписать всё заново;
ломать стабильную Tilda/YooKassa/ЧЗ схему;
вызывать внешние API на GET-страницах;
показывать токены и секреты;
логировать secret key;
менять raw_json WB;
распределять report-level WB расходы по товарам без отдельного режима;
слепо верить PASS в отчёте MiMo без проверки логики и скринов;
делать много задач в одном vXXX;
засорять интерфейс лишними блоками;
возвращать складской модуль как главную функцию;
менять логику SKU без согласования.
```
