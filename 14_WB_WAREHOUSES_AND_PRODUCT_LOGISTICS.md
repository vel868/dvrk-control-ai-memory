# 14 — WB: склады и логистика по товарам

Этот файл описывает складскую логистику Wildberries в DVRK Control: страницы `/wb/warehouses`, `/wb/warehouses/{warehouse}`, `/wb/product-warehouses`, логику “склад → товары” и “товар → склады”, поля склада, ограничения данных WB и правила расчёта средней логистики по складам.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в складской аналитике WB, логистике по складам, карточках товаров, таблицах складов и расчётах логистики на 1 товар.

---

## 1. Зачем нужна складская логистика WB

Пользователь хотел видеть не только среднюю экономику по товару, но и понимать:

```text
на каких складах товар продаётся;
где логистика дороже;
где обратная логистика выше;
как склад влияет на остаток после WB;
какой склад даёт лучшую экономику по товару.
```

Это особенно важно для товаров:

```text
massagor;
sumka600;
sumka02;
lopata0126;
другие WB-артикулы.
```

---

## 2. Главная идея

В WB-модуле есть две разные аналитические модели:

```text
склад → товары
товар → склады
```

Обе нужны.

Первая отвечает:

```text
какие товары и с какой логистикой идут через конкретный склад.
```

Вторая отвечает:

```text
через какие склады продаётся конкретный товар и где логистика дешевле/дороже.
```

---

## 3. Страница `/wb/warehouses`

Страница `/wb/warehouses` — это аналитика:

```text
склад → товары
```

Она должна показывать список складов и основные показатели.

Пример колонок:

```text
Склад
Товаров
Выкуплено
Невыкуп / обратная логистика
Цена продажи / 1
WB начислил forPay / 1
Логистика WB / 1
Обратная логистика WB / 1
Расходы WB после forPay / 1
Остаток после WB / 1
```

---

## 4. Страница `/wb/warehouses/{warehouse}`

Страница детализации конкретного склада.

Должна показывать:

```text
название склада;
период;
все товары на этом складе;
количество выкупов;
невыкупы;
среднюю цену продажи;
комиссию;
эквайринг;
forPay;
логистику;
обратную логистику;
остаток после WB;
ссылки на карточки товаров.
```

Смысл страницы:

```text
понять, какие товары на конкретном складе дают дорогую или дешёвую логистику.
```

---

## 5. Страница `/wb/product-warehouses`

Пользователь уточнил, что ему нужна не только логика “склад → товары”, но и обратная таблица:

```text
товар → все склады
```

Для этого была создана страница:

```text
/wb/product-warehouses
```

Она должна показывать каждый товар отдельным блоком, а внутри — таблицу складов.

Пример:

```text
massagor
  Электросталь
  Коледино
  Екатеринбург
  Краснодар
  Тула
```

---

## 6. Почему `/wb/product-warehouses` важна

Для продавца обычно вопрос звучит так:

```text
у меня есть конкретный товар;
на каких складах он продаётся;
где логистика дорогая;
где обратная логистика высокая;
куда лучше поставлять товар.
```

Поэтому “товар → склады” часто полезнее, чем “склад → товары”.

---

## 7. Карточка товара `/wb/products/{nm_id}`

В карточке товара должен быть блок:

```text
Логистика по складам
```

Он должен показывать:

```text
склад;
выкуплено;
невыкуп;
логистика WB / 1;
обратная логистика WB / 1;
расходы после forPay / 1;
остаток после WB / 1;
доля склада в продажах товара.
```

Так оператор видит складскую экономику прямо в карточке товара.

---

## 8. Поля склада в WB

При нормализации были добавлены поля:

```text
warehouse_name
warehouse_id
ppvz_office_name
ppvz_office_id
warehouse_source
```

Важно понимать:

```text
не все поля склада в WB заполнены одинаково;
иногда warehouseName / warehouseId пустые;
часто используется officeName;
ppvzOfficeName — это ПВЗ / офис выдачи, не всегда склад.
```

---

## 9. Проблема WB-данных по складам

По отчёту v160 было найдено:

```text
officeName заполнено примерно 41%;
officeName у redeemed_sale заполнено примерно 99%;
officeName у nonbuyout_return_to_wb заполнено примерно 99%;
officeName у logistics заполнено примерно 38%;
warehouseName заполнено 0%;
warehouseId заполнено 0%.
```

Вывод:

```text
WB не всегда даёт чистое поле склада.
Часто приходится использовать officeName как лучший доступный источник.
```

---

## 10. warehouse_source

`warehouse_source` нужен, чтобы явно понимать, из какого поля получено название склада.

Примеры источников:

```text
warehouseName
officeName
ppvzOfficeName
unknown
```

Если склад получен из `officeName`, в диагностике желательно показывать:

```text
source = officeName
```

Так другой ИИ не будет думать, что это гарантированное warehouseName.

---

## 11. Нельзя путать склад и ПВЗ

`ppvzOfficeName` часто относится к ПВЗ / офису выдачи.

Нельзя без проверки считать:

```text
ppvzOfficeName = склад WB
```

Если данных склада нет, можно использовать доступное поле как приближение, но UI/diagnostics должны быть честными.

Правильно:

```text
Склад / офис WB
Источник: officeName
```

или:

```text
Склад определён по officeName
```

## 14. Какие операции учитывать

Для складской логистики важно разделять операции:

```text
redeemed_sale — реальные выкупленные продажи;
logistics — логистические операции;
nonbuyout_return_to_wb — невыкуп / обратная логистика;
customer_return_after_redeem — возврат после выкупа;
storage — хранение;
deduction — удержания;
penalty — штраф;
additional_payment — доплаты.
```

Нельзя считать все строки как продажи.

---

## 15. Выкуплено

Количество выкупленных продаж считать по реальным продажам:

```text
classification = redeemed_sale
```

Не считать:

```text
логистику;
возмещение издержек;
хранение;
удержания;
штрафы;
невыкупы;
возвраты.
```

---

## 16. Невыкуп / обратная логистика

Невыкуп нельзя считать по `quantity`, потому что у таких строк quantity может быть 0.

Использовать:

```text
returnAmount
```

Исправленная логика:

```text
SUM(COALESCE(NULLIF(return_amount, 0), 1))
```

---

## 17. Логистика WB / 1

Средняя логистика на 1 товар по складу:

```text
Логистика WB / 1 =
  сумма прямой логистики по складу и товару / количество выкупленных продаж
```

Денежное поле:

```text
deliveryService
```

Дополнительно может учитываться:

```text
rebillLogisticCost
```

если он относится к логистике.

---

## 18. Обратная логистика WB / 1

Средняя обратная логистика:

```text
Обратная логистика WB / 1 =
  сумма обратной логистики по складу и товару / количество выкупленных продаж
```

В UI показывать как расход:

```text
− Обратная логистика WB / 1
```

---

## 19. Расходы после forPay / 1

Для склада и товара можно показывать:

```text
− Расходы WB после forPay / 1
```

Состав:

```text
прямая логистика;
обратная логистика;
rebill логистика;
хранение, если распределено;
приёмка, если распределена;
удержания, если распределены;
штрафы, если распределены;
минус доплаты.
```

Если report-level расходы не распределены, нужно показывать предупреждение.

---

## 20. Остаток после WB / 1

Формула:

```text
Остаток после WB / 1 =
  WB начислил forPay / 1
  − Логистика WB / 1
  − Обратная логистика WB / 1
  − Прочие расходы после forPay / 1
```

Важно:

```text
это не чистая прибыль;
это остаток после расходов WB.
```

---

## 21. Знаки в UI

Пользователь хочет видеть расходы со знаком минус.

Правильный вид:

```text
Цена продажи / 1: 513 ₽
− Комиссия WB / 1: −24 ₽
− Эквайринг / 1: −21 ₽
WB начислил forPay / 1: 467 ₽
− Логистика WB / 1: −106 ₽
− Обратная логистика WB / 1: −7 ₽
Остаток после WB / 1: 354 ₽
```

На складских страницах такие же правила.

---

## 22. Периоды складской аналитики

Складская аналитика должна использовать тот же период, что и WB-модуль.

Важно:

```text
если данные WB отстают, период считать от последней фактической даты операции;
не показывать пустой текущий день, если WB ещё не дал данные;
период должен совпадать между /wb, /wb/products, /wb/warehouses и /wb/product-warehouses.
```

---

## 23. Ссылки между страницами

Нужны удобные переходы:

```text
/wb → /wb/warehouses
/wb → /wb/product-warehouses
/wb/products/{nm_id} → блок складов
/wb/warehouses/{warehouse} → товары склада
/wb/product-warehouses → карточка товара
```

Пользователь должен быстро переходить от общей экономики к складам.

---

## 24. Фильтры

Полезные фильтры:

```text
период;
товар / vendorCode;
nmId;
склад;
минимальное количество выкупов;
скрыть товары без продаж;
показать только дорогую логистику;
показать только высокую обратную логистику.
```

Фильтры не должны вызывать WB API на GET. Они работают по локальной БД.

---

## 25. Сортировки

Полезные сортировки:

```text
по выкуплено;
по логистика / 1;
по обратная логистика / 1;
по расходы после forPay / 1;
по остаток после WB / 1;
по названию склада;
по товару.
```

---

## 26. Производительность

Складская аналитика может быть тяжёлой.

Нужно:

```text
агрегировать в SQL;
использовать индексы по period/date, nm_id, vendor_code, warehouse_name;
не грузить raw_json в UI без необходимости;
ограничивать количество строк;
использовать пагинацию/лимиты.
```

---

## 27. Диагностика складов

В `/wb/diagnostics` полезно показывать:

```text
процент заполненности warehouseName;
процент заполненности warehouseId;
процент заполненности officeName;
процент заполненности ppvzOfficeName;
сколько строк со складом unknown;
top warehouse_source;
примеры raw_json по складам.
```

Это нужно, чтобы понимать качество данных WB.

---

## 28. Что нельзя делать

Нельзя:

```text
считать ppvzOfficeName гарантированным складом;
скрывать warehouse_source;
считать deliveryAmount рублями;
считать returnAmount рублями;
считать quantity по логистике как продажи;
смешивать невыкуп и возврат после выкупа;
вызывать WB API на GET;
пересчитывать raw_json;
распределять report-level deduction по складам без отдельной методики.
```

---

## 29. Report-level расходы и склады

Некоторые расходы могут быть на уровне отчёта, а не склада.

Например:

```text
deduction;
paidStorage;
penalty;
paidAcceptance.
```

Если у расхода нет склада, его нельзя честно привязать к складу без методики.

Варианты:

```text
показывать как нераспределённые;
распределять по продажам — только отдельный режим;
распределять по forPay — только отдельный режим;
оставить вне складской экономики.
```

Текущее правило:

```text
не распределять report-level расходы по складам без отдельного решения.
```

---

## 30. Проверка massagor

После изменений складской логистики обязательно проверить `massagor`.

Проверки:

```text
massagor виден в /wb/product-warehouses;
у него есть несколько складов;
логистика / 1 отличается по складам;
обратная логистика / 1 отображается;
расходы со знаком минус;
остаток после WB / 1 считается логично;
ссылка на карточку товара работает.
```

---

## 31. Проверка страниц

После изменений проверить:

```text
/wb/warehouses открывается;

/wb/product-warehouses открывается;
/wb/products/{nm_id} открывается;
/wb открывается;
/wb/products открывается;
/wb/unit-economics открывается;

```

## 36. Главное резюме

Кратко:

```text
WB складская аналитика нужна в двух видах: склад → товары и товар → склады.
/wb/warehouses показывает склады.
/wb/warehouses/{warehouse} показывает товары склада.
/wb/product-warehouses показывает каждый товар и все его склады.
Карточка товара должна иметь блок "Логистика по складам".
officeName часто используется как лучший доступный источник склада.
ppvzOfficeName не всегда склад.
deliveryService — деньги, deliveryAmount не деньги.
returnAmount — количество/признак невыкупа, не деньги.
Расходы в UI показывать со знаком минус.
Report-level расходы не распределять по складам без отдельной методики.
GET /wb* не вызывает WB API.
```
