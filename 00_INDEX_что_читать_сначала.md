# 00_INDEX — что читать сначала

Этот файл — навигация по пакету handoff-документов для передачи проекта **DVRK Control** другому ИИ-агенту.

Цель пакета — не пересказать всю переписку подряд, а дать другому ИИ структурированную картину проекта: что уже сделано, как устроена система, какие бизнес-правила нельзя ломать, как пользователь работал через MiMo, какие проверки обязательны и куда двигаться дальше.

---

## 1. Что такое DVRK Control

**DVRK Control** — это FastAPI-приложение для управления операционными процессами интернет-магазина DVRK SEVER.

Система используется для:

- приёма и обработки оплаченных заказов из Tilda;
- работы с Честным Знаком;
- отправки второго / закрывающего чека через YooKassa;
- просмотра заказов оператором;
- CRM / покупательской аналитики;
- статистики продаж;
- логов, аудита и backup;
- отдельной WB-аналитики по финансовым отчётам Wildberries;
- страницы подключений Tilda / YooKassa;
- технических проверок и системной диагностики.

Проект не является абстрактным экспериментом. Это рабочая dev-система, где уже есть стабильная бизнес-логика, которую нельзя ломать.

## 3. Как пользователь работал с проектом

Пользователь не пишет код руками как программист. Работа строилась через связку:

```text
Пользователь → ChatGPT → task-файл vXXX.md → MiMo → VPS → report_vXXX.md → проверка → следующий vXXX
```

Ключевой принцип: **одна задача — один task-файл — один запуск MiMo — один отчёт — проверка**.

Другому ИИ нельзя начинать давать хаотичные команды. Нужно продолжать этот же рабочий процесс.

Подробно это описано в файле:

```text
02_WORKFLOW_как_мы_работали_через_MiMo.md
```

## 6. Самые критичные правила проекта

### 6.1 Tilda / YooKassa / Честный Знак

Стабильная схема:

```text
Tilda принимает оплату.
Tilda формирует первый чек предоплаты.
DVRK Control формирует второй закрывающий чек.
```

DVRK Control не должен формировать первый чек.

Для маркированных товаров используется суффикс SKU:

```text
ZH
```

Примеры:

```text
5006ZH       — товар требует ЧЗ
5006         — товар не требует ЧЗ
1007ZH_CLUB  — товар требует ЧЗ и является CLUB-товаром
DELIVERY     — доставка, не товар
```

Если признак расчёта `full_prepayment`, нужен закрывающий чек.

Если признак расчёта `full_payment`, второй чек обычно не нужен, и система должна предупреждать, что для ЧЗ это неправильная настройка.

---

### 6.2 Внешние API на GET-страницах

Нельзя вызывать внешние API при обычном открытии страниц.

Запрещено на GET:

```text
YooKassa API
Tilda API
WB API
```

Внешние API можно вызывать только по явному действию пользователя, например POST-кнопкой:

```text
Проверить подключение
Синхронизировать WB
Отправить чек
```

---

### 6.3 Секреты и токены

Нельзя:

```text
показывать токены в HTML;
логировать secret key;
отдавать secret key в API response;
выводить .env на страницу;
писать секреты в audit/meta/logs;
```

Secret Key должен отображаться только маской.

---

### 6.4 WB raw_json

Нельзя менять оригинальные данные WB:

```text
wb_sales_report_rows.raw_json
```

`raw_json` — источник правды. Нормализация должна идти в отдельные колонки и таблицы, но оригинал не меняется.

## 7. Как правильно продолжать разработку

Новый ИИ должен работать так:

1. Уточнить задачу.
2. Не менять сразу много разделов.
3. Создать один task-файл:

```text
/opt/dvrk_control_app/vXXX.md
```

4. В task-файле обязательно указать:

- цель;
- запреты;
- backup;
- конкретные файлы;
- проверки;
- browser / DOM проверки через Playwright;
- screenshots `/tmp/...png`;
- отчёт `report_vXXX.md`;
- git commit;
- git tag;
- удаление task-файла после успеха.

5. Пользователь запускает MiMo:

```text
Прочитай vXXX.md и выполни автономно.
```

6. MiMo создаёт отчёт.
7. Отчёт нужно проверить, а не принимать автоматически.
8. Только после проверки давать следующий task.

## 8. Что нельзя делать новому ИИ

Нельзя:

- давать пользователю десятки команд без task-файла;
- предлагать менять всё сразу;
- смешивать WB-модуль с Tilda/YooKassa/ЧЗ;
- ломать стабильную схему второго чека;
- трогать production;
- вызывать внешние API на GET;
- показывать секреты;
- верить отчёту MiMo без проверки;
- запускать старые проверки v127-v168 без причины;
- делать уведомления, Telegram или email без нового согласования пользователя;
- распределять WB report-level расходы по товарам без отдельного режима;
- считать чистую прибыль WB, пока не заведены расходы продавца.

1. Доработать `/integrations` до полноценной панели настроек YooKassa/Tilda.

## 11. Главная мысль для другого ИИ

Этот проект уже имеет рабочую стабильную логику. Главная задача нового ИИ — **продолжать аккуратно**, а не перепридумывать систему.

Правильный стиль работы:

```text
медленно;
пошагово;
через vXXX task-файлы;
с backup;
с browser-проверками;
с отчётом;
с проверкой результата;
без ломания стабильной бизнес-логики.
```
