# 12 — WB Analytics: обзор модуля Wildberries

Этот файл описывает WB-модуль проекта DVRK Control: зачем он нужен, какие страницы есть, какие API Wildberries используются, какие поля важны, как устроена синхронизация и какие ограничения нельзя нарушать.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в `/wb`, WB API, синхронизации, финансовых отчётах, расчётах по товарам, логистике, складам и unit economics.

---

## 1. Зачем нужен WB-модуль

Пользователь является селлером на Wildberries.

WB-модуль нужен, чтобы понимать:

```text
сколько WB начисляет по каждому товару;
какие расходы WB удерживает;
сколько остаётся после комиссии, эквайринга и логистики;
какая средняя экономика на 1 товар;
какие товары продаются в плюс или в минус;
какая логистика по складам;
какие корректировки WB влияют на forPay;
как позже добавить себестоимость и чистую прибыль.
```

Главная цель:

```text
понять экономику конкретного товара, например массажера massagor.
```

---

## 2. WB-модуль отдельный

WB-модуль не смешивается с Tilda / YooKassa / ЧЗ.

Отдельно:

```text
Tilda / YooKassa / ЧЗ — заказы сайта и закрывающие чеки.
WB — финансовые отчёты Wildberries и экономика маркетплейса.
```

Нельзя смешивать:

```text
WB-продажи с заказами Tilda;
WB forPay с YooKassa оплатами;
WB bankPaymentSum с заказами сайта;
WB товары с CRM покупателей сайта.
```

## 4. `/wb`

Главная страница WB Analytics.

Должна показывать:

```text
статус подключения;
последнюю синхронизацию;
период данных;
количество отчётов;
количество строк;
общие суммы;
сверку начислений;
среднюю экономику по товарам;
последние API-запросы;
быстрые переходы.
```

Главная таблица должна быть не просто списком операций, а таблицей средней экономики по товарам.

---

## 5. `/wb/sync`

Страница ручной синхронизации WB.

Должна позволять:

```text
выбрать период;
выбрать режим;
указать limit, если нужно;
запустить синхронизацию вручную;
проверить подключение;
видеть результат;
видеть ошибку API.
```

Важно:

```text
WB API не должен вызываться автоматически при открытии страницы.
Синхронизация только по ручному POST/action.
```

---

## 6. `/wb/settings`

Настройки WB API.

Пользователь хотел:

```text
вводить WB token прямо на сайте;
сохранять token;
использовать read-only токен;
не усложнять лишними мерами;
но не показывать token полностью.
```

WB token должен иметь категорию:

```text
Финансы
```

Иначе финансовые отчёты не будут доступны.

## 11. `/wb/unit-economics`

Страница средней unit economics.

Главная идея:

```text
рассчитать среднюю экономику на 1 реализованный товар.
```

Показатели:

```text
Цена продажи / 1
− Комиссия WB / 1
− Эквайринг / 1
+/- Корректировка до forPay / 1
= WB начислил forPay / 1
− Логистика WB / 1
− Обратная логистика WB / 1
− Расходы WB после forPay / 1
= Остаток после WB / 1
```

## 14. `/wb/field-dictionary`

Словарь полей WB.

Был создан, чтобы не путаться в полях отчётов.

Важная задача:

```text
зафиксировать, что означает каждое поле WB;
где поле используется;
является ли поле денежным;
является ли поле количеством;
как его показывать в UI.
```

Критичный пример:

```text
deliveryAmount и returnAmount — это не рубли.
Это количество / показатели доставки и возврата.
```

Денежные поля логистики:

```text
deliveryService
rebillLogisticCost
```

## 15. `/wb/calculation-rules`

Страница правил расчёта.

Должна объяснять:

```text
что такое retailAmount;
что такое ppvzSalesCommission;
что такое acquiringFee;
что такое forPay;
что такое bankPaymentSum;
что такое deliveryService;
что такое paidStorage;
что такое deduction;
что такое additionalPayment;
как считается остаток после WB.
```

Это нужно, чтобы пользователь и другой ИИ не путали финансовые понятия.

## 16. `/wb/money-flow`

Должна объяснять:

```text

получается forPay;

точный банковский перевод на 1 товар недоступен напрямую.
```

## 20. Какой WB API использовать

Для точной финансовой аналитики использовать финансовые отчёты WB.

Основной метод:

```text
POST /api/finance/v1/sales-reports/detailed
```

Для списка отчётов:

```text
POST /api/finance/v1/sales-reports/list
```

Старый метод `reportDetailByPeriod` не должен быть основной опорой для новой логики.

---

## 21. Почему не orders/sales

Оперативные WB orders/sales не подходят для точной финансовой экономики.

Причина:

```text
там нет полной финансовой детализации;
не видно всех удержаний;
не видно report-level расходов;
не хватает forPay / bankPaymentSum логики;
можно неправильно посчитать прибыль.
```

Для денег использовать финансовые отчёты реализации.

---

## 22. rrdId

Финансовый detailed API работает через `rrdId`.

Схема:

```text
первый запрос: rrdId = 0;
получить пачку строк;
взять последний rrdId;
следующий запрос — с этим rrdId;
продолжать, пока API не вернёт 204 или пустой результат.
```

Важно:

```text
не терять rrdId;
не создавать дубли;
сохранять raw_json.
```

---

## 23. Token category

WB token должен иметь категорию:

```text
Финансы
```

Если токен без финансов, API финансовых отчётов не будет работать.

В UI нужно показывать понятную ошибку:

```text
Токен WB не имеет доступа к категории Финансы.
```

---

## 24. Rate limit и cooldown

WB API нельзя спамить.

Пользователь хотел:

```text
минимум 10 минут;
лучше 1 час;
только ручная синхронизация;
не вызывать API при открытии страниц.
```

Правило:

```text
GET /wb* не вызывает WB API.
```

Только POST/action на синхронизацию.

---

## 25. Raw JSON

WB raw_json нужно сохранять.

Причина:

```text
WB меняет поля;
часть полей может быть не нормализована;
нужна диагностика;
можно пересчитать данные позже;
можно найти скрытые корректировки.
```

Нельзя удалять raw_json или менять его смысл без отдельной миграции.

## 28. forPay

`forPay` — это сумма, которую WB начислил продавцу по товарной строке после части удержаний.

Официальный смысл в проекте зафиксирован так:

```text
WB начислил продавцу за реализованный товар.
```

Важно:

```text
forPay — не чистая прибыль;
forPay — не финальный банковский перевод;
forPay уже учитывает комиссию и эквайринг;
после forPay ещё могут быть логистика, хранение, удержания, штрафы.
```

---

## 29. bankPaymentSum

`bankPaymentSum` — это сумма на уровне финансового отчёта.

Смысл:

```text
итого к оплате / фактическая сумма отчёта;
report-level показатель.
```

Важно:

```text
bankPaymentSum нельзя точно распределить на 1 товар без отдельной методики;
не считать bankPaymentSum / qty как точную выплату по каждому товару.
```

---

## 30. Основная формула отчёта

Упрощённо:

```text
bankPaymentSum ≈ forPaySum
                 − deliveryServiceSum
                 − paidStorageSum
                 − paidAcceptanceSum
                 − deductionSum
                 − penaltySum
                 + additionalPaymentSum
```

Эквайринг уже учтён в forPay и не должен вычитаться повторно после forPay.

---

## 31. Операции WB

В отчётах есть разные операции.

Не все строки с quantity относятся к реальным продажам.

Важные типы:

```text
Продажа / Продажа
Возврат / Возврат
Логистика
Возмещение издержек по перевозке
Хранение
Удержание
Коррекция логистики
Штраф
Компенсация скидки по программе лояльности
Добровольная компенсация при возврате
Возмещение за выдачу и возврат товаров на ПВЗ
```

---

## 32. Классификация операций

Принятая классификация:

```text
Продажа / Продажа → redeemed_sale
Логистика / возмещения → logistics
Возврат / Возврат → customer_return_after_redeem
Хранение → storage
Штраф → penalty
Удержание → deduction
Компенсация лояльности → additional_payment
```

Главное:

```text
не считать все строки с quantity как продажи.
```

---

## 33. Невыкуп / обратная логистика

Была ошибка: невыкуп считался как 0, потому что quantity = 0.

Правильная логика:

```text
для nonbuyout использовать returnAmount;
если returnAmount > 0 — считать как невыкуп / обратная логистика.
```

Пример исправления:

```text
SUM(COALESCE(NULLIF(return_amount, 0), 1))
```

---

## 34. Периоды WB

Для WB важно считать периоды от последней фактической даты операции, если данные WB отстают.

Пример:

```text
загрузка до 03.07.2026;
последняя дата операции в строках 28.06.2026;
период 7 дней считается от 28.06.2026, а не от текущей даты.
```

Это нужно, чтобы не показывать пустые периоды.

---

## 35. Сверка отчётов и строк

На `/wb` и `/wb/reports` важна сверка:

```text
forPaySum по отчётам;
forPay по товарным строкам;
bankPaymentSum по отчётам;
расчётный bankPayment по строкам;
разницы.
```

Разница может быть нормальной, если:

```text
есть report-level расходы;
есть операции не на уровне товара;
есть корректировки;
часть расходов не распределена.
```

Нужно показывать разницу честно, а не прятать.

## 36. Текущие важные данные massagor

По отдельному XLSX и handoff ранее было зафиксировано для массажера `massagor`:

```text
WB начисляет по строкам продаж примерно 462.16 ₽ за 1 реализованный массажер.
После логистики WB и возмещения издержек остаётся примерно 310.16 ₽ на 1 проданный массажер.
```

Важно:

```text
310 ₽ — это остаток после WB, не чистая прибыль.
Чистая прибыль считается позже после расходов продавца:
закупка, доставка до WB, фулфилмент, упаковка, маркировка, реклама/ДРР, налог.
```

## 37. Что ещё нужно добавить позже


```text

```

## 38. Что нельзя ломать в WB-модуле

Нельзя:

```text
вызывать WB API на GET;
логировать WB token;
показывать WB token полностью;
удалять raw_json;
считать deliveryAmount рублями;
считать returnAmount рублями;
считать все quantity как продажи;
повторно вычитать эквайринг после forPay;
называть forPay чистой прибылью;
распределять report-level deduction по товарам без отдельного режима;
смешивать WB с Tilda/YooKassa/ЧЗ.
```
