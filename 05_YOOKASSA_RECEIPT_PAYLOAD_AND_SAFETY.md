# 05 — YooKassa: payload закрывающего чека и безопасность

Этот файл описывает, как в проекте DVRK Control должен формироваться второй / закрывающий чек через YooKassa, какие поля payload использовать, как работать с Честным Знаком, как защищаться от дублей и что категорически нельзя ломать.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в YooKassa-интеграции, `/chz`, `/orders/{id}`, отправке чеков, receipt status, idempotency и polling.

---

## 1. Главный принцип

В стабильной архитектуре проекта:

```text
Tilda принимает оплату.
Tilda / YooKassa формируют первый чек предоплаты.
DVRK Control формирует только второй закрывающий чек.
```

DVRK Control не должен формировать первый чек оплаты.

Второй чек отправляется только после того, как заказ готов к закрытию:

```text
для обычного товара — когда понятно, что ЧЗ не нужен;
для маркированного товара — когда оператор ввёл код Честного Знака;
для смешанного заказа — когда все ZH-позиции имеют коды ЧЗ.
```

---

## 2. Когда закрывающий чек нужен

Ориентироваться нужно на признак расчёта.

```text
full_prepayment  → закрывающий чек нужен
full_payment     → закрывающий чек обычно не нужен
unknown          → нельзя отправлять, нужно проверить
```

Правило:

```text
Если Tilda / YooKassa уже прислали full_payment,
DVRK Control не должен отправлять второй чек.
```

Если признак расчёта неизвестен:

```text
показать оператору "Проверить признак расчёта";
заблокировать отправку закрывающего чека;
не пытаться угадать.
```

---

## 3. Когда закрывающий чек можно отправлять

Минимальные условия:

```text
заказ существует;
есть YooKassa payment_id;
суммы закрывающего чека сходятся с фактической оплатой;
payment_mode = full_prepayment;
receipt ещё не отправлен успешно;
нет pending/succeeded дубликата;
для всех ZH-позиций введён код ЧЗ;
обычные товары не требуют кода ЧЗ;
оператор подтвердил отправку.
```

Для UI:

```text
кнопка отправки чека должна быть disabled,
пока условия не выполнены.
```

Для backend:

```text
все условия нужно проверять повторно на сервере,
даже если UI уже заблокировал кнопку.
```

---

## 4. Защита от дублей

Это критично.

Перед отправкой второго чека нужно проверить:

```text
нет receipt_id со статусом succeeded;
нет receipt_id со статусом pending;
нет активной попытки отправки;
не был ли уже использован idempotency key;
не отправлялся ли чек для этого заказа ранее.
```

Если чек уже succeeded:

```text
не отправлять повторно;
показать оператору статус;
дать ссылку/receipt_id, если есть.
```

Если чек pending:

```text
не отправлять повторно;
запустить или продолжить polling статуса.
```

Если предыдущая попытка failed:

```text
разрешить повтор только осознанно;
показать причину ошибки;
использовать новый idempotency key, если это корректно для сценария.
```

---

## 5. Idempotency key

Для отправки чека через YooKassa нужен idempotency key.

Рекомендуемый смысл ключа:

```text
closing_receipt:{order_id}:{attempt_number}
```

Или другая стабильная схема проекта, если она уже реализована.

Главное:

```text
один и тот же logical request не должен случайно уйти дважды;
повторная отправка должна быть контролируемой.
```

Idempotency key не должен содержать секреты или персональные данные.

---

## 6. Polling статуса чека

После отправки чека YooKassa может вернуть статус не сразу.

Разрешён polling только для чеков:

```text
sent
pending
```

Нельзя делать polling на обычном GET page load без причины.

Допустимая схема:

```text
оператор открыл карточку чека;
система видит pending;
JS периодически спрашивает backend;
backend проверяет YooKassa;
после succeeded/failed polling останавливается.
```

Примерный интервал из текущей логики:

```text
10 секунд
```

---

## 7. ФФД и маркировка

Для маркированных товаров используется ФФД 1.2.

Для Честного Знака YooKassa ожидает:

```json
"mark_code_info": {
  "gs_1m": "..."
}
```

Важно:

```text
использовать mark_code_info.gs_1m;
не использовать другие поля mark_code_info без необходимости;
не кодировать код ЧЗ в Base64;
передавать код как есть в формате, который ожидает YooKassa;
добавлять mark_code_info только к ZH-позициям.
```

---

## 8. SKU и Честный Знак

В DVRK Control признак ЧЗ определяется через SKU.

```text
Если SKU содержит ZH → товар требует ЧЗ.
Если SKU не содержит ZH → товар не требует ЧЗ.
```

Примеры:

```text
5006ZH       → нужен ЧЗ
5006         → ЧЗ не нужен
1007ZH_CLUB  → нужен ЧЗ
1007_CLUB    → ЧЗ не нужен
DELIVERY     → доставка, ЧЗ не нужен
```

Нельзя:

```text
добавлять mark_code_info товару без ZH;
требовать код ЧЗ для DELIVERY;
требовать код ЧЗ для обычных товаров;
использовать XH вместо ZH.
```

---

## 9. Структура items во втором чеке

Закрывающий чек должен включать все позиции заказа:

```text
товары с ЧЗ;
товары без ЧЗ;
доставку.
```

Но `mark_code_info` добавляется только для маркированных товаров.

Пример логики items:

```text
ZH-товар:
  payment_subject = marked
  payment_mode = full_payment
  mark_code_info.gs_1m = код ЧЗ

Обычный товар:
  payment_subject = commodity
  payment_mode = full_payment
  mark_code_info отсутствует

Доставка:
  payment_subject = service
  payment_mode = full_payment
  mark_code_info отсутствует
```

---

## 10. payment_subject

Использовать:

```text
marked     — маркированный товар с ЧЗ
commodity  — обычный товар
service    — доставка
```

Правила:

```text
ZH-товар → marked
не-ZH товар → commodity
доставка → service
```

---

## 11. payment_mode

Для товарных позиций в закрывающем чеке использовать:

```text
payment_mode = full_payment
```

Это именно закрывающий чек, поэтому позиции закрываются полным расчётом.

Не путать с первым чеком Tilda, где была предоплата.

---

## 12. settlements

В закрывающем чеке использовать settlement, который отражает, что оплата была ранее внесена как предоплата.

Стабильная схема:

```json
"settlements": [
  {
    "type": "prepayment",
    "amount": {
      "value": "...",
      "currency": "RUB"
    }
  }
]
```

Сумма settlement должна соответствовать сумме чека.

---

## 13. Суммы в закрывающем чеке

Закрывающий чек должен строиться по суммам после скидки.

Финансовая модель заказа:

```text
goods_gross_sum      — товары до скидки
goods_discount_sum   — скидка / промокод
goods_net_sum        — товары после скидки
delivery_sum         — доставка
order_total_paid     — фактически оплачено
```

Формула:

```text
goods_net_sum + delivery_sum = order_total_paid
```

В чек идут:

```text
товары после распределения скидки;
доставка;
итог = фактически оплачено.
```

Если сумма не сходится:

```text
не отправлять чек;
показать warning;
записать в audit/log.
```

---

## 14. Распределение скидки

Скидка распределяется только между товарами.

Доставка не участвует.

Формула:

```text
item_discount = goods_discount_sum * item_gross / goods_gross_sum
item_net_sum = item_gross - item_discount
```

Если товар один:

```text
вся скидка относится к нему.
```

После распределения могут быть копейки округления. Их нужно обработать так, чтобы итог закрывающего чека сходился с оплатой.

---

## 15. Пример закрывающего чека

Условный заказ:

```text
Товар 1007ZH:
  gross: 1000.00
  discount: 100.00
  net: 900.00
  ЧЗ: нужен

Косметичка 5010:
  gross: 299.00
  discount: 29.90
  net: 269.10
  ЧЗ: не нужен

Доставка:
  500.00

Итого оплачено:
  1669.10
```

Items:

```text
1007ZH:
  amount = 900.00
  payment_subject = marked
  payment_mode = full_payment
  mark_code_info.gs_1m = введённый код

5010:
  amount = 269.10
  payment_subject = commodity
  payment_mode = full_payment

DELIVERY:
  amount = 500.00
  payment_subject = service
  payment_mode = full_payment
```

Settlements:

```text
type = prepayment
amount = 1669.10
```

## 16. Preview перед отправкой

Перед реальной отправкой закрывающего чека оператор должен видеть preview:

```text
товары;
количество;
суммы до скидки;
скидка;
суммы после скидки;
доставка;
итог чека;
итог оплаты;
ЧЗ-коды;
YooKassa payment_id;
payment_mode;
предупреждения.
```

Если всё корректно:

```text
оператор ставит checkbox подтверждения;
после этого кнопка отправки активна.
```

Если есть проблемы:

```text
кнопка disabled;
показать конкретную причину.
```

---

## 17. Confirm перед отправкой

Перед отправкой чека нужно показывать confirm-окно.

Смысл confirm:

```text
Оператор понимает, что сейчас будет отправлен закрывающий чек в YooKassa.
```

Даже если это dev/test режим, UI должен быть дисциплинированным.

Backend всё равно обязан повторно проверить безопасность.

---

## 18. Audit events

Нужно писать audit events для важных действий:

```text
открыт preview;
открыт confirm;
оператор подтвердил отправку;
отправка заблокирована backend;
чек отправлен;
YooKassa вернула pending;
YooKassa вернула succeeded;
YooKassa вернула failed;
получена ошибка API;
попытка повторной отправки заблокирована.
```

В audit нельзя писать секреты.

---

## 19. Ошибки YooKassa

При ошибках нужно показывать оператору:

```text
понятный статус;
текст ошибки, если безопасно;
receipt_id, если он есть;
что делать дальше.
```

Не нужно показывать raw secret, headers авторизации или полный sensitive payload.

---

## 20. Что нельзя делать на GET

GET-страницы не должны вызывать внешние API.

Запрещено на GET:

```text
создавать чек;
отправлять чек;
проверять YooKassa подключение;
делать произвольный запрос к YooKassa;
дергать Tilda API;
дергать WB API.
```

Разрешено на GET:

```text
читать локальную БД;
показывать последний известный статус;
показывать сохранённый receipt_id;
показывать локальную историю.
```

---

## 21. Страница /chz/orders/{id}

Это рабочее место оператора для закрывающего чека.

Там должно быть:

```text
информация о заказе;
покупатель;
доставка;
оплата;
позиции;
нужен ли ЧЗ;
поля ввода ЧЗ;
preview закрывающего чека;
статус YooKassa;
кнопка отправки;
audit/log событий.
```

Кнопка отправки должна быть активна только если:

```text
чек нужен;
суммы сходятся;
все ZH-коды введены;
нет succeeded/pending дубликата;
оператор подтвердил действие.
```

---

## 22. Обычный заказ без ЧЗ

Для заказа без ZH:

```text
ЧЗ-код не нужен;
mark_code_info отсутствует;
если full_prepayment — отправляется закрывающий чек без ЧЗ;
если full_payment — закрывающий чек не нужен.
```

Обычный товар не должен блокироваться из-за отсутствия ЧЗ.

---

## 23. Смешанный заказ

Для смешанного заказа:

```text
часть позиций может требовать ЧЗ;
часть позиций не требует ЧЗ;
доставка не требует ЧЗ.
```

Проверка готовности:

```text
все ZH-позиции имеют коды;
обычные позиции не требуют кодов;
итог чека сходится;
payment_mode подходит.
```

---

## 24. Проверки после изменений

Если другой ИИ меняет YooKassa / ЧЗ, обязательно проверить:

```text
python -m py_compile ...
./scripts/smoke_test.sh
/orders
/chz
/chz/orders/{id}
/logs
/integrations
```

И специальные сценарии:

```text
ZH + full_prepayment + нет кода → кнопка disabled;
ZH + full_prepayment + код есть → можно отправлять;
не-ZH + full_prepayment → можно отправлять без ЧЗ;
full_payment → второй чек не нужен;
unknown → проверка признака расчёта;
повторная отправка succeeded чека заблокирована;
pending чек не отправляется повторно;
GET-страницы не вызывают YooKassa API;
secret key не виден в HTML/API/logs.
```
