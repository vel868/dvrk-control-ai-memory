# 08 — Заказы и ЧЗ: рабочий процесс оператора

Этот файл описывает рабочий процесс оператора в DVRK Control: страницы `/orders`, `/orders/{id}`, `/chz`, `/chz/orders/{id}`, статусы заказов, ввод кодов Честного Знака, отправку второго закрывающего чека и контроль скидок/доставки.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в заказах, ЧЗ, карточке заказа, карточке закрывающего чека, статусах, таблицах, кнопках отправки чека и логике оператора.

---

## 1. Главная задача раздела заказов

Раздел заказов нужен, чтобы оператор видел оплаченные заказы из Tilda и понимал, что с ними делать дальше.

Основные действия оператора:

```text
открыть очередь заказов;
найти нужный заказ;
проверить покупателя, доставку, оплату и товары;
понять, нужен ли Честный Знак;
перейти в рабочее место ЧЗ;
ввести код ЧЗ, если он нужен;
подтвердить отправку закрывающего чека;
проконтролировать статус YooKassa.
```

---

## 2. Главная схема

```text
Tilda webhook
↓
DVRK Control создаёт заказ
↓
Заказ появляется в /orders
↓
Если есть ZH-товар — появляется задача в /chz
↓
Оператор открывает /chz/orders/{id}
↓
Вводит код ЧЗ или подтверждает, что ЧЗ не нужен
↓
DVRK Control формирует preview закрывающего чека
↓
Оператор подтверждает отправку
↓
YooKassa получает второй закрывающий чек
↓
Статус чека обновляется
```

---

## 3. Страница /orders

`/orders` — это очередь заказов.

Она должна показывать заказы компактно, но с достаточным количеством информации.

Основные колонки:

```text
ID
Удалить / скрыть
Tilda ID
Покупатель
Телефон
Email
Адрес
Доставка
Статус
ЧЗ
Позиции
Сумма
Дата
Действия
```

---

## 4. Покупатель в таблице заказов

Для отображения покупателя использовать приоритет:

```text
delivery_fio
↓
customer_name
↓
пусто
```

В SQL/логике это соответствует идее:

```sql
COALESCE(NULLIF(delivery_fio, ''), customer_name, '')
```

Это важно, потому что ФИО получателя доставки часто точнее, чем общее имя покупателя.

---

## 5. Доставка в заказах

В интерфейсе нельзя упрощать доставку до просто:

```text
Доставка
```

Пользователь хочет видеть способ доставки.

Примеры:

```text
Почта России — отделение
CDEK доставка до ПУНКТА ВЫДАЧИ
```

Доставка:

```text
не требует ЧЗ;
не является товаром;
в чеке идёт как service;
не участвует в распределении скидки.
```

---

## 6. Статусы в /orders

Статусы должны быть понятными.

Примеры:

```text
Новый заказ
Оплачен
Нужен код ЧЗ
Готов к закрывающему чеку
Закрывающий чек отправлен
Закрывающий чек succeeded
Чек не нужен
Проверить признак расчёта
Ошибка YooKassa
```

Не показывать только технические значения без объяснения.

---

## 7. ЧЗ-колонка в /orders

Колонка ЧЗ должна быстро отвечать на вопрос:

```text
нужен ли Честный Знак;
введён ли код;
можно ли отправлять закрывающий чек;
чек не нужен или уже отправлен.
```

Примеры отображения:

```text
Нужен код ЧЗ
Код ЧЗ введён
ЧЗ не нужен
Чек не нужен
Проверить признак расчёта
```

---

## 8. Действия в /orders

Основные действия:

```text
Открыть заказ
Открыть ЧЗ / закрывающий чек
Удалить / скрыть тестовый заказ
```

Опасные действия должны иметь confirm.

Пример:

```text
Удалить заказ?
Это действие уберёт заказ из рабочей очереди. Данные аудита могут сохраниться.
```

---

## 9. Live update /orders

В проекте есть live-update для `/orders`.

Важно:

```text
не ломать автообновление;
не делать внешние API вызовы при автообновлении;
обновлять только локальные данные;
не сбрасывать фильтры оператора.
```

---

## 10. Карточка заказа /orders/{id}

Карточка заказа должна быть структурирована блоками:

```text
Покупатель
Доставка
Оплата
YooKassa
Товары / позиции заказа
Быстрые действия
Аудит / события
```

Оператор должен быстро понять:

```text
кто покупатель;
как доставлять;
что оплачено;
какие товары;
какой YooKassa payment_id;
нужен ли ЧЗ;
что делать дальше.
```

---

## 11. Финансовый блок карточки заказа

Финансы должны отображаться прозрачно:

```text
Товары до скидки
Скидка / промокод
Товары после скидки
Доставка
Итого оплачено
```

Это критично для закрывающего чека.

Формула:

```text
Товары после скидки + Доставка = Итого оплачено
```

Если формула не сходится:

```text
показать warning;
не давать отправить закрывающий чек автоматически;
записать проблему в audit/log.
```

---

## 12. Таблица товаров в карточке заказа

Таблица товаров должна показывать:

```text
SKU
Название
Количество
Цена до скидки
Скидка
Цена после скидки
Нужен ЧЗ
Код ЧЗ, если есть
Статус
```

Если есть промокод, должна быть видна строка или блок:

```text
Скидка / промокод
```

---

## 13. Распределение скидки

Скидка распределяется только между товарами.

Доставка не участвует.

Формула:

```text
item_discount = goods_discount_sum * item_gross / goods_gross_sum
item_net_sum = item_gross - item_discount
```

Если товар один, вся скидка относится к нему.

---

## 14. Страница /chz

`/chz` — это очередь заказов, где оператор видит задачи по Честному Знаку и закрывающим чекам.

Страница должна показывать:

```text
заказы, где нужен код ЧЗ;
заказы, где закрывающий чек нужен без ЧЗ;
заказы, где нужно проверить признак расчёта;
статусы YooKassa;
покупателя;
доставку;
товары;
сумму;
действие.
```

---

## 15. Статусы на /chz

Основные статусы:

```text
Нужен код ЧЗ
Готов к закрывающему чеку
Закрывающий чек без ЧЗ
Чек не нужен
Проверить признак расчёта
Чек отправлен
Ошибка отправки
```

Логика:

```text
ZH + full_prepayment + нет кода → Нужен код ЧЗ
ZH + full_prepayment + код есть → Готов к закрывающему чеку
full_prepayment без ZH → Закрывающий чек без ЧЗ
full_payment → Чек не нужен
unknown → Проверить признак расчёта
```

---

## 16. Страница /chz/orders/{id}

Это рабочее место оператора для второго закрывающего чека.

На странице должно быть:

```text
информация о заказе;
покупатель;
доставка;
оплата;
YooKassa payment_id;
товары;
нужен ли ЧЗ;
поля ввода кодов ЧЗ;
решение "ЧЗ не нужен", если применимо;
preview закрывающего чека;
сверка сумм;
кнопка отправки;
история действий.
```

---

## 17. Ввод кода ЧЗ

Код ЧЗ нужен только для товаров, у которых SKU содержит:

```text
ZH
```

Примеры:

```text
5006ZH
1007ZH_CLUB
2008ZH
```

Для таких товаров оператор должен ввести код.

Без кода закрывающий чек не должен отправляться.

---

## 18. Обычные товары без ЧЗ

Если SKU не содержит `ZH`, код ЧЗ не нужен.

Примеры:

```text
5006
1007_CLUB
5010
DELIVERY
```

Обычные товары не должны блокировать отправку закрывающего чека из-за отсутствия ЧЗ.

---

## 19. Смешанные заказы

В одном заказе могут быть ZH и non-ZH товары.

Правило:

```text
все ZH-позиции должны иметь код ЧЗ;
обычные позиции не требуют код;
доставка не требует код;
закрывающий чек включает все позиции.
```

`mark_code_info` добавляется только к ZH-позициям.

---

## 20. Решение "ЧЗ не нужен"

Для обычных товаров может быть состояние:

```text
ЧЗ не нужен
```

Но это не должно использоваться для ZH-товаров как обход обязательного кода.

Нельзя разрешать оператору скрыть требование ЧЗ для ZH-позиции без отдельного строгого сценария.

---

## 21. Preview закрывающего чека

Перед отправкой чек должен показываться оператору.

Preview должен включать:

```text
позиции чека;
SKU;
названия;
количество;
суммы после скидок;
доставку;
итог чека;
итог оплаты;
payment_subject;
payment_mode;
mark_code_info для ZH;
settlements;
warnings.
```

Если чек не сходится с оплатой, кнопка отправки disabled.

---

## 22. Сверка сумм

Главная сверка:

```text
итог товаров после скидки + доставка = фактически оплачено
```

И:

```text
итог закрывающего чека = фактически оплачено
```

Если есть расхождение:

```text
показать предупреждение;
записать в audit/log;
заблокировать отправку.
```

---

## 23. Кнопка отправки закрывающего чека

Кнопка должна быть активна только если:

```text
payment_mode = full_prepayment;
закрывающий чек нужен;
нет succeeded/pending чека;
все ZH-коды введены;
суммы сходятся;
operator confirmation выполнен.
```

Если кнопка disabled, должна быть причина.

Примеры причин:

```text
Введите код ЧЗ для товара 1007ZH.
Сумма чека не совпадает с оплатой.
Признак расчёта unknown.
Чек уже отправлен.
YooKassa receipt pending.
```

---

## 24. Confirm перед отправкой

Перед отправкой нужен confirm.

Смысл:

```text
оператор осознанно подтверждает отправку второго закрывающего чека.
```

Можно использовать checkbox + confirm dialog.

Backend всё равно должен повторно проверить условия.

---

## 25. YooKassa statuses

Нужно показывать:

```text
не отправлен;
pending;
succeeded;
failed;
не нужен;
ошибка;
```

Если pending:

```text
не отправлять повторно;
можно polling статуса;
интервал примерно 10 секунд;
остановить polling после succeeded/failed.
```

---

## 26. Защита от повторной отправки

Нельзя отправлять чек повторно, если:

```text
receipt succeeded;
receipt pending;
есть активная попытка;
idempotency key уже использован для этой операции.
```

Повтор после failed — только осознанно и с понятным audit.

---

## 27. Audit / история действий

Все важные действия нужно писать в audit/log:

```text
заказ получен от Tilda;
дубликат webhook;
код ЧЗ введён;
код ЧЗ изменён;
preview открыт;
confirm открыт;
отправка подтверждена;
отправка заблокирована;
чек отправлен;
YooKassa pending;
YooKassa succeeded;
YooKassa failed;
ошибка API;
повторная отправка заблокирована.
```

Не логировать секреты.

---

## 28. Что нельзя делать на GET

На GET-страницах нельзя:

```text
отправлять чек;
проверять YooKassa через внешний API;
создавать новые внешние запросы;
вызывать Tilda API;
вызывать WB API.
```

GET может только читать локальную БД.

---

## 29. Удаление / скрытие заказов

В проекте была добавлена кнопка удаления заказа, в основном для очистки тестовых заказов.

Важно:

```text
опасное действие должно иметь confirm;
не удалять аудит без отдельного решения;
не ломать реальные данные;
показывать понятное предупреждение.
```

---

## 30. Статистика и заказы

Данные заказов используются в статистике и CRM.

Поэтому при изменениях в заказах нельзя ломать:

```text
/stats;
/customers;
финансовые суммы;
скидки;
доставку;
товарные позиции;
секции SKU.
```

---

## 31. Покупатели / CRM

Покупатели агрегируются из заказов.

Нельзя:

```text
определять пол по имени;
склеивать покупателей только по имени;
отправлять маркетинг без согласия;
ломать email/phone.
```

Подробнее в файле:

```text
09_STATS_CUSTOMERS_CRM.md
```

---

## 32. ЧЗ и интерфейс

Для ЧЗ-страниц важно:

```text
оператор не должен гадать;
статус должен быть понятный;
если действие недоступно — причина видна;
подсказки через tooltip;
не делать огромные постоянные блоки, если достаточно tooltip.
```

---

## 33. Проверки после изменений

Если другой ИИ меняет `/orders` или `/chz`, нужно проверить:

```text
/orders открывается;
/orders/{id} открывается;
/chz открывается;
/chz/orders/{id} открывается;
/stats открывается;
/customers открывается;
/logs открывается;
/integrations открывается;
GET не вызывает YooKassa API;
GET не вызывает Tilda API;
WB API не вызывается.
```

Сценарии:

```text
ZH + full_prepayment + нет кода → кнопка disabled;
ZH + код → можно готовить чек;
non-ZH + full_prepayment → чек без ЧЗ;
full_payment → чек не нужен;
unknown → проверить признак расчёта;
скидка/доставка сходятся с оплатой;
повторная отправка чека заблокирована.
```

## 36. Главное резюме

Кратко:

```text
/orders — очередь заказов.
/orders/{id} — карточка заказа.
/chz — очередь задач по ЧЗ и закрывающим чекам.
/chz/orders/{id} — рабочее место оператора.
ZH в SKU означает, что нужен ЧЗ.
full_prepayment означает, что закрывающий чек нужен.
full_payment означает, что закрывающий чек не нужен.
Обычные товары не требуют ЧЗ.
Доставка не требует ЧЗ.
Перед отправкой чека нужны preview, сверка сумм и confirm.
Повторная отправка чека должна быть заблокирована.
GET-страницы не вызывают внешние API.
```
