# 04 — Tilda / YooKassa / Честный Знак: главная бизнес-логика

Этот файл объясняет главную стабильную логику проекта DVRK Control: как система принимает заказы из Tilda, как работает YooKassa, когда нужен второй закрывающий чек, и как определяется необходимость Честного Знака.

Файл предназначен для другого ИИ-агента, который будет продолжать разработку проекта. Его нужно читать до любых изменений в `/orders`, `/chz`, YooKassa, Tilda webhook, логике чеков и SKU.

---

## 1. Главная схема проекта

DVRK Control не является интернет-магазином. Интернет-магазин и оплата находятся на Tilda.

Стабильная схема такая:

```text
Покупатель оформляет заказ на Tilda
↓
Tilda принимает оплату через YooKassa
↓
Tilda / YooKassa формируют первый чек предоплаты 100%
↓
Tilda отправляет webhook в DVRK Control
↓
DVRK Control сохраняет заказ
↓
Оператор проверяет заказ и Честный Знак
↓
DVRK Control отправляет второй закрывающий чек через YooKassa
```

Критично:

```text
DVRK Control НЕ должен формировать первый чек.
DVRK Control должен формировать только второй / закрывающий чек.
```

---

## 2. Почему нужен второй закрывающий чек

Для заказов через Tilda используется схема предоплаты.

Первый чек:

```text
Тип: предоплата 100%
Формирует: Tilda / YooKassa
Момент: когда покупатель оплатил заказ
```

Второй чек:

```text
Тип: полный расчёт / закрывающий чек
Формирует: DVRK Control
Момент: после обработки заказа
```

Для товаров с маркировкой Честного Знака второй чек особенно важен, потому что туда должен попасть код маркировки.

---

## 3. Роли сервисов

### 3.1 Tilda

Tilda отвечает за:

```text
каталог товаров;
оформление заказа;
приём оплаты;
первый чек предоплаты;
отправку webhook после оплаты.
```

Tilda не должна использоваться DVRK Control как источник постоянной синхронизации при каждом открытии страниц. Webhook — основной источник заказов.

### 3.2 YooKassa

YooKassa отвечает за:

```text
оплату;
чеки;
API для отправки второго закрывающего чека.
```

В DVRK Control YooKassa используется только там, где это действительно нужно:

```text
отправка второго закрывающего чека;
проверка статуса чека;
polling статуса уже отправленного чека;
ручная проверка подключения через POST.
```

Запрещено:

```text
вызывать YooKassa API на GET-страницах;
отправлять чек при простом открытии страницы;
делать повторную отправку закрывающего чека без защиты от дублей.
```

### 3.3 DVRK Control

DVRK Control отвечает за:

```text
приём webhook от Tilda;
сохранение заказа;
определение, нужен ли ЧЗ;
рабочее место оператора;
ввод кода ЧЗ;
решение "ЧЗ не нужен" для обычных товаров;
отправку второго закрывающего чека;
аудит действий;
логи;
статистику;
CRM;
WB-аналитику отдельно.
```

## 4. Ключевые поля webhook Tilda

В webhook Tilda важны такие поля:

```text
payment[orderid]
payment[systranid]
payment[amount]
payment[subtotal]
payment[delivery_price]
payment[delivery_fio]
Email
Phone
payment[products][i][sku]
payment[products][i][name]
payment[products][i][quantity]
payment[products][i][price]
payment[products][i][amount]
```

Особенно важные поля:

```text
payment[orderid]      — ID заказа Tilda / внешний ID
payment[systranid]    — payment_id YooKassa
payment[amount]       — фактически оплаченная сумма
payment[products]     — товары заказа
sku                   — главный источник логики ЧЗ и секций
```

`payment[systranid]` используется как связка с YooKassa payment_id.

## 5. Логика признака расчёта

Главные значения:

```text
full_prepayment  — предоплата 100%
full_payment     — полный расчёт
unknown          — неизвестно / нужно проверить
```

Правила:

```text
Если full_prepayment:
  закрывающий чек нужен.

Если full_payment:
  закрывающий чек обычно не нужен.

Если unknown:
  нельзя автоматически отправлять закрывающий чек.
  Нужно показывать оператору "Проверить признак расчёта".
```

Критично:

```text
Если Tilda/YooKassa уже прислали full_payment, DVRK Control не должен отправлять второй чек как будто была предоплата.
```

---

## 6. Логика Честного Знака через SKU

В проекте принято простое и стабильное правило:

```text
Если SKU содержит суффикс ZH — товар требует Честный Знак.
Если SKU не содержит ZH — товар не требует Честный Знак.
```

Примеры:

```text
5006ZH       — товар требует ЧЗ
5006         — товар не требует ЧЗ
1007ZH_CLUB  — товар требует ЧЗ и участвует в CLUB-акции
1007_CLUB    — товар без ЧЗ, но CLUB
DELIVERY     — доставка, не товар и не требует ЧЗ
```

Важно:

```text
Использовать только ZH.
Не использовать XH.
```

---

## 7. Base SKU

Для товаров с ЧЗ нужно уметь получать базовый SKU.

Пример:

```text
5006ZH       → base_sku = 5006
1007ZH_CLUB  → base_sku = 1007_CLUB или 1007 в зависимости от места применения
```

Главный смысл:

```text
ZH — технический признак маркировки.
Он не должен ломать аналитику товара.
```

---

## 8. Смешанные заказы

В одном заказе могут быть:

```text
товары с ЧЗ;
товары без ЧЗ;
доставка;
CLUB-товары;
товары со скидкой / промокодом.
```

Правило:

```text
Закрывающий чек должен включать все позиции заказа.
Но mark_code_info добавляется только к позициям, которые требуют ЧЗ.
```

Пример:

```text
Футболка 1007ZH — нужен mark_code_info
Косметичка 5010 — mark_code_info не нужен
Доставка DELIVERY — mark_code_info не нужен
```

---

## 9. Статусы по заказу и ЧЗ

Упрощённая логика:

```text
ZH + full_prepayment + нет кода ЧЗ
  → нужен код ЧЗ

ZH + full_prepayment + код ЧЗ введён
  → готов к закрывающему чеку

full_prepayment без ZH
  → закрывающий чек нужен, но ЧЗ не нужен

full_payment
  → закрывающий чек не нужен

unknown payment_mode
  → проверить признак расчёта
```

---

## 10. Обычные товары без ЧЗ

Если товар не содержит `ZH`, оператор не должен вводить код ЧЗ.

Для таких заказов:

```text
если payment_mode = full_prepayment:
  нужен закрывающий чек без mark_code_info

если payment_mode = full_payment:
  закрывающий чек не нужен
```

Решение "ЧЗ не нужен" должно быть доступно только там, где это логически нужно, и не должно блокировать обычные товары.

---

## 11. Заказы с ЧЗ

Если товар содержит `ZH`:

```text
оператор должен ввести код ЧЗ;
код должен попасть во второй закрывающий чек;
без кода нельзя отправлять закрывающий чек для маркированной позиции.
```

Код ЧЗ не должен кодироваться в Base64. В YooKassa используется `mark_code_info.gs_1m`.

Подробности payload находятся в отдельном файле:

```text
05_YOOKASSA_RECEIPT_PAYLOAD_AND_SAFETY.md
```

## 12. Скидки и промокоды

В Tilda могут быть промокоды и скидки.

Стабильная финансовая модель заказа:

```text
goods_gross_sum      — товары до скидки
goods_discount_sum   — скидка / промокод
goods_net_sum        — товары после скидки
delivery_sum         — доставка
order_total_paid     — фактически оплачено
```

Формула:

```text
goods_net_sum + delivery_sum = order_total_paid
```

Скидка распределяется только между товарами, доставка не участвует.

Пропорциональное распределение:

```text
item_discount = goods_discount_sum * item_gross / goods_gross_sum
item_net_sum = item_gross - item_discount
```

Для одного товара вся скидка уходит на него.

---

## 13. Что должно попадать во второй чек

Во второй закрывающий чек должны попадать суммы после скидок.

Пример:

```text
Товар до скидки: 299.00
Скидка: 29.90
Товар после скидки: 269.10
Доставка: 1053.03
Итого оплачено: 1322.13
```

Закрывающий чек должен сходиться с фактической оплатой.

Критично:

```text
Если сумма закрывающего чека не сходится с оплатой — не отправлять чек автоматически.
```

---

## 14. Доставка

Доставка в заказе не является товаром.

Она не требует ЧЗ.

Но в интерфейсе важно показывать не просто "Доставка", а понятный способ:

```text
Почта России — отделение
CDEK доставка до ПУНКТА ВЫДАЧИ
```

Пользователь хочет видеть, как именно доставляется заказ.

## 15. CLUB и секции товаров

В Tilda используются секции и SKU-правила:

```text
1xxx = Мужское
2xxx = Женское
5xxx = Аксессуары
ZH   = Честный Знак
CLUB = клубная / акционная механика
```

Примеры:

```text
1007_CLUB     = Мужское + CLUB
1007ZH_CLUB   = Мужское + ЧЗ + CLUB
2008_CLUB     = Женское + CLUB
2008ZH_CLUB   = Женское + ЧЗ + CLUB
5010_CLUB     = Аксессуары + CLUB
5006ZH        = Аксессуары + ЧЗ
DELIVERY      = Доставка
```

Важно:

```text
Не считать ошибкой, если пользователь положил кепку, жилетку или другой товар в аксессуары.
Система не должна спорить с каталогом пользователя.
```

---

## 16. Что нельзя ломать

Критичные запреты:

```text
Не делать первый чек в DVRK Control.
Не отправлять второй чек при full_payment.
Не отправлять чек без кода ЧЗ для ZH-товара.
Не добавлять mark_code_info обычным товарам.
Не добавлять mark_code_info доставке.
Не вызывать YooKassa API на GET.
Не менять правило ZH без отдельного решения.
Не менять настройки Tilda webhook без понимания последствий.
Не ломать распределение скидок.
Не смешивать WB-логику с Tilda/YooKassa/ЧЗ.
```

---

## 17. Как другому ИИ проверять себя

Если задача касается заказов, ЧЗ, Tilda или YooKassa, нужно обязательно проверить:

```text
/orders открывается;
/orders/{id} открывается;
/chz открывается;
/chz/orders/{id} открывается;
обычный заказ без ZH не требует кода ЧЗ;
ZH-заказ требует код ЧЗ;
full_payment не даёт отправлять второй чек;
full_prepayment даёт закрывающий чек;
суммы после скидок сходятся;
GET-страницы не вызывают YooKassa API;
секреты YooKassa не выводятся в HTML/logs.
```

## 18. Текущая точка проекта

На момент этого handoff:

```text
Заказы из Tilda принимаются.
ЧЗ-логика через ZH работает.
Второй закрывающий чек YooKassa реализован.
Скидки / промокоды учитываются.
CRM и статистика работают.
Логи и backup есть.
Страница /integrations создана, но её ещё можно дорабатывать.
WB-модуль вынесен отдельно и не должен смешиваться с Tilda/YooKassa.
```

Последний важный tag перед этим handoff:

```text
v1.65-logs-integrations-settings
```
