# 06 — Tilda webhook: стабильные настройки и поля

Этот файл описывает, как в проекте DVRK Control должен быть настроен webhook Tilda, какие поля Tilda передаёт в систему, какие настройки считаются стабильными, и что нельзя менять без отдельного решения.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в Tilda webhook, `/tilda`, `/integrations`, обработке заказов, товарах, доставке, оплате, скидках и промокодах.

---

## 1. Роль Tilda в проекте

Tilda — это внешний сайт / интернет-магазин, где покупатель оформляет и оплачивает заказ.

Tilda отвечает за:

```text
каталог товаров;
страницы товаров;
корзину;
оформление заказа;
приём оплаты через YooKassa;
первый чек предоплаты;
отправку webhook в DVRK Control после оплаты.
```

DVRK Control не должен заменять Tilda как витрину. DVRK Control принимает уже оплаченный заказ и дальше помогает оператору обработать его.

---

## 2. Главная стабильная схема

Рабочая схема:

```text
Покупатель оформляет заказ на Tilda
↓
Покупатель оплачивает заказ через YooKassa
↓
Tilda / YooKassa формирует первый чек предоплаты 100%
↓
Tilda отправляет webhook в DVRK Control
↓
DVRK Control сохраняет заказ
↓
DVRK Control определяет ЧЗ по SKU
↓
Оператор вводит код ЧЗ, если нужен
↓
DVRK Control отправляет второй закрывающий чек
```

Критично:

```text
Tilda должна отправлять webhook только после оплаты.
DVRK Control не должен принимать неоплаченные заказы как готовые к обработке.
```

---

## 3. Стабильные настройки webhook Tilda

В Tilda должны быть выставлены такие настройки:

```text
API METHOD: Off
Передавать данные по товарам в заказе — массивом: Включено
Отправлять только после оплаты: Включено
application/json: Выключено
Cookies: Выключено
externalid: Выключено
```

Эти настройки считаются стабильными для текущего проекта.

---

## 4. Почему API METHOD должен быть Off

В стабильной схеме не нужен Tilda API METHOD.

Tilda должна просто отправлять webhook с данными заказа.

Если включить API METHOD или поменять формат отправки, структура данных может измениться, и DVRK Control может перестать правильно читать:

```text
товары;
оплату;
доставку;
промокоды;
YooKassa payment_id;
ЧЗ-признак через SKU.
```

Поэтому:

```text
API METHOD должен быть Off.
```

---

## 5. Почему товары должны передаваться массивом

Важная настройка:

```text
Передавать данные по товарам в заказе — массивом: Включено
```

DVRK Control ожидает товары в структуре:

```text
payment[products][0][sku]
payment[products][0][name]
payment[products][0][quantity]
payment[products][0][price]
payment[products][0][amount]

payment[products][1][sku]
payment[products][1][name]
...
```

Если товары не передаются массивом, система может потерять:

```text
SKU;
количество;
цены;
названия товаров;
признак ZH;
CLUB-признак;
товарные суммы.
```

---

## 6. Почему webhook только после оплаты

Настройка:

```text
Отправлять только после оплаты: Включено
```

Это важно, потому что DVRK Control строит рабочую очередь по оплаченным заказам.

Если Tilda отправит webhook до оплаты, в системе могут появиться:

```text
неоплаченные заказы;
заказы без YooKassa payment_id;
заказы без корректного признака расчёта;
заказы, по которым нельзя формировать закрывающий чек.
```

Поэтому:

```text
Tilda webhook должен приходить только после оплаты.
```

---

## 7. Почему application/json выключен

В текущей стабильной схеме webhook читается в формате form-urlencoded / обычной структуры Tilda.

Настройка:

```text
application/json: Выключено
```

Если включить JSON, структура payload может измениться, и текущий parser может перестать правильно читать поля.

Менять это можно только отдельной задачей с полной проверкой parser.

---

## 8. Почему Cookies выключены

Настройка:

```text
Cookies: Выключено
```

Cookies в webhook не нужны для бизнес-логики проекта.

Они могут добавить шум, лишние данные и риск случайного хранения ненужной информации.

---

## 9. Почему externalid выключен

Настройка:

```text
externalid: Выключено
```

В текущей схеме DVRK Control использует свои стабильные поля:

```text
payment[orderid]
payment[systranid]
```

Включение externalid может изменить ожидания по идентификаторам и вызвать путаницу.

---

## 10. Ключевые поля webhook

Главные поля, которые DVRK Control должен читать из webhook:

```text
payment[orderid]
payment[systranid]
payment[amount]
payment[subtotal]
payment[delivery_price]
payment[delivery_fio]
Email
Phone
Name
payment[products]
```

---

## 11. payment[orderid]

Поле:

```text
payment[orderid]
```

Смысл:

```text
ID заказа Tilda / внешний номер заказа.
```

Используется для:

```text
поиска заказа;
отображения Tilda ID;
связи с webhook;
защиты от дублей;
аудита.
```

---

## 12. payment[systranid]

Поле:

```text
payment[systranid]
```

Смысл:

```text
YooKassa payment_id
```

Это очень важное поле.

Оно используется для:

```text
связи заказа с платежом YooKassa;
отправки второго закрывающего чека;
проверки статуса оплаты / чека;
отображения YooKassa-блока в карточке заказа.
```

Если `payment[systranid]` отсутствует, закрывающий чек нельзя отправлять автоматически.

---

## 13. payment[amount]

Поле:

```text
payment[amount]
```

Смысл:

```text
фактически оплаченная сумма заказа.
```

Используется как итоговая контрольная сумма.

Закрывающий чек должен сходиться с этой суммой.

---

## 14. payment[subtotal]

Поле:

```text
payment[subtotal]
```

Смысл может зависеть от Tilda-настроек, но обычно это сумма товаров до доставки.

Его нельзя использовать вслепую как единственный источник правды.

Нужно сверять:

```text
товарные строки;
скидки;
доставку;
payment[amount].
```

---

## 15. payment[delivery_price]

Поле:

```text
payment[delivery_price]
```

Смысл:

```text
стоимость доставки.
```

Доставка должна попадать в заказ отдельной строкой или отдельным полем.

В чеке доставка:

```text
payment_subject = service
mark_code_info отсутствует
```

Доставка не требует Честный Знак.

---

## 16. payment[delivery_fio]

Поле:

```text
payment[delivery_fio]
```

Смысл:

```text
ФИО получателя доставки.
```

В UI проекта при отображении покупателя используется приоритет:

```text
delivery_fio
↓
customer_name
↓
пусто
```

То есть если есть `delivery_fio`, он должен быть основным отображаемым ФИО.

---

## 17. Email и Phone

Поля:

```text
Email
Phone
```

Используются для:

```text
карточки заказа;
CRM / покупателей;
поиска;
склейки покупателей;
связи с клиентом.
```

Важно:

```text
Не определять пол покупателя по имени.
Не использовать email/phone в UI-настройках.
Не логировать лишние персональные данные без необходимости.
```

---

## 18. payment[products]

Товарные строки — ключевой источник данных.

Ожидаемые поля:

```text
payment[products][i][sku]
payment[products][i][name]
payment[products][i][quantity]
payment[products][i][price]
payment[products][i][amount]
```

Используются для:

```text
создания позиций заказа;
определения ЧЗ через SKU;
расчёта скидок;
расчёта закрывающего чека;
статистики;
CRM;
секции товара;
CLUB-признака.
```

---

## 19. SKU — главный технический признак

SKU используется не только как артикул, но и как источник правил.

Текущие правила:

```text
1xxx = Мужское
2xxx = Женское
5xxx = Аксессуары
ZH   = Честный Знак
CLUB = клуб / акция
DELIVERY = доставка
```

Примеры:

```text
1007_CLUB     = Мужское + CLUB
1007ZH_CLUB   = Мужское + ЧЗ + CLUB
2008ZH        = Женское + ЧЗ
5010_CLUB     = Аксессуары + CLUB
5006ZH        = Аксессуары + ЧЗ
DELIVERY      = Доставка
```

---

## 20. Честный Знак через SKU

Главное правило:

```text
Если SKU содержит ZH — товар требует код ЧЗ.
Если SKU не содержит ZH — ЧЗ не нужен.
```

Нельзя использовать `XH`.

Нельзя требовать ЧЗ для доставки.

Нельзя добавлять `mark_code_info` для обычных товаров.

---

## 21. CLUB

`CLUB` в SKU означает клубную / акционную механику.

Примеры:

```text
1007_CLUB
1007ZH_CLUB
5010_CLUB
```

`CLUB` не означает ЧЗ.

ЧЗ определяется только через `ZH`.

---

## 22. Доставка

Доставка может приходить в полях Tilda отдельно.

В интерфейсе важно отображать не просто слово "Доставка", а понятный способ, если он есть:

```text
Почта России — отделение
CDEK доставка до ПУНКТА ВЫДАЧИ
```

Доставка:

```text
не товар;
не требует ЧЗ;
в чеке идёт как service;
не участвует в распределении скидки.
```

---

## 23. Скидки и промокоды

Tilda может передавать скидку / промокод.

В системе принята модель:

```text
goods_gross_sum      — товары до скидки
goods_discount_sum   — скидка / промокод
goods_net_sum        — товары после скидки
delivery_sum         — доставка
order_total_paid     — фактически оплачено
```

Формула:

```text
goods_net_sum + delivery_sum = order_total_paid
```

Скидка распределяется только между товарами.

Доставка не участвует в скидке.

---

## 24. Пример скидки

Пример заказа:

```text
Товар: 299.00
Промокод: 29.90
Товар после скидки: 269.10
Доставка: 1053.03
Итого оплачено: 1322.13
```

В карточке заказа и чеке должно быть понятно:

```text
товары до скидки;
скидка / промокод;
товары после скидки;
доставка;
итого оплачено.
```

---

## 25. Защита от дублей webhook

Tilda может повторно отправить webhook.

DVRK Control должен защищаться от дублей.

Основные признаки для dedupe:

```text
payment[orderid]
payment[systranid]
внутренний order_id
```

Если webhook повторный:

```text
не создавать дубль заказа;
не отправлять повторные уведомления;
не ломать существующие позиции;
записать событие в audit/log.
```

---

## 26. Что делать при неполном webhook

Если не хватает важных полей:

```text
нет payment[orderid];
нет payment[systranid];
нет товаров;
нет суммы;
```

Система должна:

```text
сохранить raw webhook, если такая логика есть;
показать warning;
не отправлять закрывающий чек;
не пытаться угадать данные;
записать audit/log.
```

---

## 27. Tilda webhook и первый чек

Важно:

```text
Первый чек формирует Tilda / YooKassa.
DVRK Control не должен пытаться формировать первый чек.
```

DVRK Control должен понимать, что пришедший оплаченный заказ уже имеет первый чек предоплаты.

Дальше DVRK Control формирует только второй закрывающий чек, если он нужен.

---

## 28. Где показывать настройки Tilda

На момент handoff есть страница:

```text
/integrations
```

Там должны быть настройки подключения Tilda:

```text
Webhook URL;
Webhook token / secret, если используется;
критические настройки;
инструкция по настройке Tilda;
проверка webhook;
история изменений.
```

Старая страница `/tilda`, если она есть, не должна ломаться. Можно добавить на неё ссылку:

```text
Настройки подключения перенесены в /integrations
```

---

## 29. Что должно быть на странице /integrations для Tilda

Минимальный набор:

```text
Webhook URL с кнопкой копирования;
Webhook token / secret, замаскирован;
Project / Site ID, если используется;
Form ID, если используется;
кнопка "Сохранить настройки";
кнопка "Проверить webhook";
кнопка "Открыть инструкцию";
блок "Критические настройки".
```

Критические настройки должны показывать:

```text
API METHOD: Off
Передавать товары массивом: Включено
Отправлять только после оплаты: Включено
application/json: Выключено
Cookies: Выключено
externalid: Выключено
```

---

## 30. Безопасность Tilda settings

Если есть webhook token / secret:

```text
не показывать полностью в HTML;
не отдавать полностью через API;
не логировать;
не писать в audit meta;
показывать только masked;
если поле пустое при сохранении — не затирать старое значение.
```

---

## 31. Проверка webhook Tilda

Проверка webhook должна быть безопасной.

GET `/integrations` не должен вызывать внешний Tilda API.

Кнопка "Проверить webhook" может делать POST на backend и проверять:

```text
есть ли webhook URL;
есть ли token, если используется;
есть ли критические настройки;
есть ли последние webhook-события;
нет ли ошибок последних webhook.
```

Без необходимости не нужно обращаться к внешнему Tilda API.

---

## 32. Что нельзя менять без отдельной задачи

Нельзя:

```text
переводить webhook на JSON без обновления parser;
включать application/json;
отключать товары массивом;
отключать "только после оплаты";
менять правило ZH;
менять payment[systranid] как payment_id;
смешивать Tilda-заказы с WB;
автоматически отправлять чек по приходу webhook без operator flow;
вызывать Tilda API на GET-странице.
```

---

## 33. Проверки после изменений

Если другой ИИ меняет Tilda webhook или настройки Tilda, нужно проверить:

```text
/tilda открывается или корректно ссылается на /integrations;
/integrations открывается;
Webhook URL отображается;
secret/token замаскирован;
GET /integrations не вызывает Tilda API;
новый webhook корректно создаёт заказ;
повторный webhook не создаёт дубль;
товары читаются массивом;
SKU читается правильно;
ZH определяется правильно;
скидка распределяется правильно;
доставка не требует ЧЗ;
/orders показывает заказ;
/chz показывает правильный статус;
закрывающий чек не отправляется автоматически на GET.
```
