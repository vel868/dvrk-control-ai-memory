# 13 — WB: путь денег и unit economics

Этот файл подробно описывает финансовую логику Wildberries в DVRK Control: что такое retailAmount, комиссия, эквайринг, корректировка до forPay, forPay, логистика, обратная логистика, bankPaymentSum, остаток после WB и почему это ещё не чистая прибыль.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в `/wb`, `/wb/products`, `/wb/products/{nm_id}`, `/wb/unit-economics`, `/wb/money-flow`, формулах, колонках, знаках расходов и расчётах прибыли.

---

## 1. Главная цель unit economics WB

Главная задача WB-модуля:

```text
понять, сколько денег остаётся на 1 товар после начислений и удержаний Wildberries.
```

Но важно разделять:

```text
Остаток после WB
и
Чистая прибыль продавца
```

Это разные вещи.

---

## 2. Главная финансовая цепочка

Упрощённо путь денег по одному товару:

```text
Цена продажи покупателю
− Комиссия WB
− Эквайринг / платёжные услуги
+/- Корректировки до forPay
= WB начислил forPay

− Логистика WB
− Обратная логистика WB
− Хранение
− Приёмка
− Удержания
− Штрафы
+ Доплаты
= Остаток после WB
```

После этого ещё нужно вычитать внешние расходы продавца:

```text
закупка;
доставка до WB;
фулфилмент;
упаковка;
маркировка;
реклама / ДРР;
налог;
прочие расходы.
```

Только после этого получается чистая прибыль.

---

## 3. Главное правило терминов

Нельзя называть `forPay` чистой прибылью.

Нельзя называть `Остаток после WB` чистой прибылью.

Правильные термины:

```text
forPay = WB начислил продавцу по товарной строке;
Остаток после WB = forPay минус расходы WB после forPay;
Чистая прибыль = остаток после WB минус расходы продавца.
```

---

## 4. retailAmount

`retailAmount` — это сумма продажи / розничная сумма по строке отчёта WB.

В UI можно показывать как:

```text
Цена продажи
Цена продажи / 1
```

Но нужно учитывать:

```text
это фактическая сумма из отчёта WB;
она может отличаться от плановой цены на сайте;
у товара могут быть скидки WB, СПП, промо и другие механики.
```

---

## 5. ppvzSalesCommission

`ppvzSalesCommission` — комиссия WB.

В UI для продавца это расход.

Показывать со знаком минус:

```text
− Комиссия WB / 1
```

Не показывать как положительную сумму без контекста, потому что пользователь должен сразу видеть, что WB забирает деньги.

---

## 6. acquiringFee

`acquiringFee` — платёжные услуги / эквайринг.

В UI это расход:

```text
− Эквайринг / 1
```

Важно:

```text
эквайринг уже учтён в forPay;
после forPay его повторно вычитать нельзя.
```

---

## 7. forPay

`forPay` — ключевое поле.

Смысл для UI:

```text
WB начислил forPay
WB начислил forPay / 1
```

Объяснение:

```text
Это сумма, которую WB начислил продавцу по товарной строке после комиссии и эквайринга, с учётом корректировок до forPay.
```

Важно:

```text
forPay — не финальный банковский перевод;
forPay — не чистая прибыль;
после forPay WB ещё удерживает логистику, хранение, приёмку, штрафы, удержания и т.д.
```

---

## 8. Базовая формула forPay

На простых товарах формула близка к:

```text
forPay ≈ retailAmount − ppvzSalesCommission − acquiringFee
```

Но после анализа было найдено, что иногда есть дополнительные корректировки.

Расширенная формула:

```text
forPay ≈ retailAmount
         − ppvzSalesCommission
         − acquiringFee
         +/− correction_to_forpay
```

---

## 9. Корректировка до forPay

Было найдено, что у некоторых товаров `forPay` не сходится только с ценой, комиссией и эквайрингом.

Пример по `lopata0126`:

```text
Цена продажи / 1: 1212 ₽
Комиссия / 1: −150 ₽
Эквайринг / 1: −48 ₽
Ожидаемый forPay: 1014 ₽
Фактический forPay / 1: 1346 ₽
Разница: +333 ₽
```

Причина:

```text
SPP / Seller Promotion Program;
ppvzReward;
paymentProcessing = "Компенсация платёжных услуг";
другие корректировки до forPay.
```

Поэтому в UI была добавлена строка:

```text
+/- Корректировка до forPay / 1
```

---

## 10. SPP и другие корректировки

В raw_json были найдены поля:

```text
spp
ppvzReward
cashbackDiscount
kvw
kvwBase
vw
vwNds
```

Часть этих полей помогает объяснять расхождения до forPay.

Важно:

```text
не игнорировать корректировки;
не подгонять формулу вручную;
сохранять raw_json;
показывать расхождение честно, если оно не объяснено.
```

---

## 11. deliveryService

`deliveryService` — денежное поле логистики WB.

Это расход после forPay.

В UI показывать:

```text
− Логистика WB / 1
```

Важно:

```text
deliveryService — рубли;
deliveryAmount — не рубли.
```

---

## 12. deliveryAmount

`deliveryAmount` — не денежное поле.

Это количество / показатель доставки.

Нельзя:

```text
считать deliveryAmount рублями;
добавлять deliveryAmount в расходы WB;
показывать deliveryAmount как ₽.
```

Если используется, то только как количественный показатель.

---

## 13. returnAmount

`returnAmount` — не денежное поле.

Используется для невыкупов / обратной логистики.

Нельзя:

```text
считать returnAmount рублями;
добавлять returnAmount как денежный расход.
```

Для подсчёта количества невыкупов:

```text
если returnAmount > 0, это признак обратного движения / невыкупа.
```

---

## 14. Обратная логистика WB

Обратная логистика — расход, связанный с невыкупом / возвратным движением.

В UI показывать:

```text
− Обратная логистика WB / 1
```

Количество невыкупов считать не по `quantity`, потому что у таких строк quantity может быть 0.

Исправленная логика:

```text
SUM(COALESCE(NULLIF(return_amount, 0), 1))
```

---

## 15. rebillLogisticCost

`rebillLogisticCost` — денежная корректировка логистики.

Её нужно учитывать в логистических расходах, если она есть.

В UI можно показывать отдельно или включать в:

```text
− Логистика WB / 1
```

Но в детализации желательно показать источник:

```text
deliveryService
rebillLogisticCost
```

---

## 16. paidStorage

`paidStorage` — хранение.

Это расход после forPay.

Показывать:

```text
− Хранение / 1
```

или внутри:

```text
− Расходы WB после forPay / 1
```

Если хранение report-level и не распределено по товарам, нужно показывать это честно.

---

## 17. paidAcceptance

`paidAcceptance` — платная приёмка.

Это расход после forPay.

Показывать как:

```text
− Приёмка / 1
```

Если report-level, не распределять по товарам без отдельного режима.

---

## 18. deduction

`deduction` — удержания WB.

Это критичное поле.

В данных было найдено крупное удержание:

```text
deduction около 340 591.60 ₽
```

Важно:

```text
если deduction report-level, нельзя автоматически размазывать по товарам без отдельного решения;
если не учесть deduction вообще, товарная экономика может быть завышена;
если размазать неправильно, экономика товара будет искажена.
```

---

## 19. penalty

`penalty` — штрафы.

Показывать как расход:

```text
− Штрафы
```

Если штраф report-level, не распределять без отдельного режима.

---

## 20. additionalPayment

`additionalPayment` — доплаты / компенсации.

В UI показывать со знаком плюс:

```text
+ Доплаты
```

В формуле после forPay это увеличивает остаток.

---

## 21. bankPaymentSum

`bankPaymentSum` — report-level показатель.

Смысл:

```text
итого к оплате по финансовому отчёту / фактическая сумма на уровне отчёта.
```

Важно:

```text
bankPaymentSum не является точной выплатой за конкретный товар;
bankPaymentSum нельзя напрямую делить на количество товара и считать точной unit economics;
bankPaymentSum нужен для сверки отчёта.
```

---

## 22. Формула report-level

Упрощённая формула отчёта:

```text
bankPaymentSum ≈ forPaySum
                 − deliveryServiceSum
                 − paidStorageSum
                 − paidAcceptanceSum
                 − deductionSum
                 − penaltySum
                 + additionalPaymentSum
```

Важное правило:

```text
Эквайринг уже учтён в forPay.
Не вычитать acquiringFee второй раз после forPay.
```

---

## 23. Остаток после WB

Остаток после WB — это управленческий показатель.

Формула:

```text
Остаток после WB =
  forPay
  − Логистика WB
  − Обратная логистика WB
  − Хранение
  − Приёмка
  − Удержания
  − Штрафы
  + Доплаты
```

Но если часть расходов report-level и не распределена по товарам, нужно показывать предупреждение.

---

## 24. Остаток после WB / 1

Для товара:

```text
Остаток после WB / 1 =
  Остаток после WB по товару / количество выкупленных продаж
```

Количество должно быть именно по реальным выкупленным продажам, а не по всем строкам.

Использовать классификацию операций.

---

## 25. Реальные продажи / выкуплено

Нельзя считать все строки с quantity как продажи.

Правильная база продаж:

```text
sellerOperName / docTypeName = Продажа / Продажа
классификация = redeemed_sale
```

`Логистика`, `Возмещение издержек`, `Удержание`, `Хранение` и другие операции не являются продажами.

---

## 26. Возвраты после выкупа

Отдельно учитывать:

```text
customer_return_after_redeem
```

Это не то же самое, что невыкуп / обратная логистика.

Нельзя смешивать:

```text
возврат после выкупа
и
невыкуп
```

---

## 27. Невыкуп

Невыкуп / обратная логистика считается отдельно.

Пример из принятых данных:

```text
massagor 30d:
  Выкуплено: 177
  Невыкуп / обратная логистика: 24
```

Важно:

```text
quantity может быть 0;
использовать returnAmount.
```

---

## 28. Пример massagor из интерфейса

После исправлений для `massagor` 30d было зафиксировано:

```text
Выкуплено: 177
Невыкуп: 24
Цена продажи / 1: 513 ₽
− Комиссия WB / 1: −24 ₽
− Эквайринг / 1: −21 ₽
WB начислил forPay / 1: 467 ₽
− Логистика WB / 1: −106 ₽
− Обратная логистика WB / 1: −7 ₽
Расходы WB после forPay / 1: −113 ₽
Остаток после WB / 1: 354 ₽
```

Это не чистая прибыль.

По отдельному XLSX `Еженедельный детализированный отчет №765190021_250093386 - 1.xlsx` было зафиксировано:

## 30. Внешние расходы продавца

Чтобы получить чистую прибыль, нужно вычесть расходы продавца.

Для massagor ранее использовались:

```text
закупка: 70 ₽
логистика до WB: 34 ₽
фулфилмент: 17 ₽
итого себестоимость: 121 ₽
ДРР / реклама: 105 ₽
налог / резерв: отдельно
```

---

## 31. Пример чистой прибыли massagor

По предыдущему handoff:

```text
Остаток после WB: 310.16 ₽
Себестоимость: 121 ₽
Реклама / ДРР: 105 ₽
```

До налога:

```text
310.16 − 121 − 105 = 84.16 ₽
```

С налогом / резервом:

```text
при УСН 6% от фактической реализации WB: примерно 54.32 ₽
при управленческом резерве 7%: примерно 49.35 ₽
если ошибочно налог 49 ₽: примерно 35.16 ₽
```

Важно:

```text
налоги — отдельная тема;
не смешивать налоговую логику с WB unit economics без отдельного решения.
```

---

## 32. Почему пользователь ожидал 70–90 ₽

Пользователь ожидал прибыль около 70–90 ₽ по массажеру.

Это не противоречит WB-цифрам, потому что:

```text
310 ₽ — это остаток после WB;
после этого надо вычесть закупку, логистику до WB, фулфилмент, рекламу, налог.
```

Именно после этих расходов получается диапазон, близкий к ожиданиям пользователя.

---

## 33. Как показывать деньги в UI

Пользователь хочет видеть расходы со знаком минус.

Правильный вид:

```text
Цена продажи / 1: 513 ₽
− Комиссия WB / 1: −24 ₽
− Эквайринг / 1: −21 ₽
+/- Корректировка до forPay / 1: +4 ₽
= WB начислил forPay / 1: 467 ₽
− Логистика WB / 1: −106 ₽
− Обратная логистика WB / 1: −7 ₽
= Остаток после WB / 1: 354 ₽
```

Не показывать расходы как положительные числа без минуса.

---

## 34. Расходы WB после forPay / 1

Это агрегированный показатель:

```text
Расходы WB после forPay / 1 =
  логистика
  + обратная логистика
  + rebill логистика
  + хранение
  + приёмка
  + удержания
  + штрафы
  − доплаты
```

В UI его можно показывать как:

```text
− Расходы WB после forPay / 1
```

Если есть расшифровка — лучше показывать отдельно.

---

## 35. Что делать с report-level расходами

Report-level расходы нельзя просто незаметно размазывать по товарам.

Варианты:

```text
оставить как нераспределённые;
показать отдельным блоком;
добавить режим распределения;
распределять по выручке / forPay / количеству только после явного решения.
```

Пока правило:

```text
не распределять report-level deduction/storage/penalty без отдельного режима.
```

---

## 36. Сверка forPay и bankPayment

На `/wb` должна быть сверка:

```text
Начислено по отчётам / forPaySum
Расчётно по товарным строкам
Разница forPay

Итого к оплате по отчётам / bankPaymentSum
Расчётно по формуле
Разница bankPayment
```

Разницы не прятать.

## 38. Что нельзя делать

Нельзя:

```text
называть forPay чистой прибылью;
называть остаток после WB чистой прибылью;
считать deliveryAmount рублями;
считать returnAmount рублями;
считать все quantity продажами;
повторно вычитать acquiringFee после forPay;
скрывать корректировку до forPay;
игнорировать deduction;
распределять report-level расходы без отдельного решения;
смешивать WB и Tilda-заказы;
вызывать WB API на GET.
```
