# 02_WORKFLOW — как мы работали через MiMo

Этот файл объясняет рабочий процесс проекта DVRK Control: как пользователь, ChatGPT и MiMo вместе вносили изменения в систему на VPS.

Главная цель документа — чтобы другой ИИ не начал работать хаотично, не давал пользователю длинные случайные команды и не ломал уже стабильную систему.

---

## 1. Что такое MiMo в этом проекте

MiMo — это ИИ-агент-исполнитель, которого пользователь запускал для работы на сервере. Пользователь сам не программирует вручную и не хочет разбирать десятки файлов руками.

Рабочая схема была такая:

```text
Пользователь объясняет задачу
→ ChatGPT формирует подробный task-файл vXXX.md
→ пользователь копирует bash-блок в SSH
→ на сервере создаётся /opt/dvrk_control_app/vXXX.md
→ пользователь говорит MiMo: "Прочитай vXXX.md и выполни автономно"
→ MiMo меняет код, запускает проверки, создаёт report_vXXX.md
→ пользователь присылает отчёт и скриншоты
→ ChatGPT проверяет результат
→ задача принимается или создаётся следующий корректирующий vXXX
```

MiMo в этом процессе — не стратег, а исполнитель. Стратегия, ограничения и проверки должны быть заранее описаны в task-файле.

---

## 2. Почему использовались task-файлы vXXX.md

Формат работы был специально выбран простой и контролируемый:

```text
1 задача = 1 task-файл = 1 запуск MiMo = 1 report-файл
```

Task-файлы создавались в рабочей папке проекта:

```text
/opt/dvrk_control_app
```

Примеры:

```text
/opt/dvrk_control_app/v160.md
/opt/dvrk_control_app/v167.md
/opt/dvrk_control_app/v168.md
```

Отчёты MiMo создавались рядом:

```text
/opt/dvrk_control_app/report_v160.md
/opt/dvrk_control_app/report_v167.md
/opt/dvrk_control_app/report_v168.md
```

Такой подход нужен, чтобы:

- пользователь мог просто скопировать один блок в SSH;
- MiMo не делал лишнего;
- задача была ограничена по объёму;
- другой ИИ понимал контекст;
- можно было проверить результат;
- в git оставалась понятная история;
- при проблеме можно было сделать следующий корректирующий task-файл.

---

## 3. Как правильно давать пользователю задание

Пользователь не хочет получать абстрактные инструкции типа: "откройте файл, найдите функцию, вставьте код".

Правильный формат ответа пользователю:

1. Коротко объяснить цель шага.
2. Дать один bash-блок, который создаёт файл `/opt/dvrk_control_app/vXXX.md`.
3. Дать короткую команду для MiMo.

Пример команды для MiMo:

```text
Прочитай vXXX.md и выполни автономно.
```

Или:

```text
Прочитай /opt/dvrk_control_app/vXXX.md и выполни автономно. В конце создай report_vXXX.md.
```

Не нужно давать пользователю длинные ручные инструкции, если можно оформить задачу для MiMo.

---

## 4. Как должен выглядеть хороший task-файл

Хороший task-файл должен быть подробным, конкретным и безопасным.

Структура:

```text
# vXXX — название задачи

Рабочая папка:
/opt/dvrk_control_app

Цель:
...

Запреты:
...

Перед началом:
...

Backup:
...

Задача 1.
...

Задача 2.
...

Проверки:
...

Документация:
...

Отчёт:
...

Git:
...

Финальный ответ:
...
```

Обязательно указывать:

- рабочую папку;
- предыдущий ожидаемый git tag;
- backup перед изменениями;
- что можно менять;
- что запрещено менять;
- какие файлы создать/изменить;
- какие проверки выполнить;
- какие screenshots сделать;
- какой report создать;
- какой commit/tag поставить.

## 6. Backup перед задачей

В task-файлах почти всегда требовался backup БД.

Для основной БД:

```bash
mkdir -p /opt/backups
pg_dump -h 127.0.0.1 -U dvrk_app dvrk_control_dev > /opt/backups/pre_vXXX_task_name_$(date +%Y%m%d_%H%M%S).sql
```

Для WB-БД, если задача касалась Wildberries:

```bash
mkdir -p /opt/backups
pg_dump -h 127.0.0.1 -U dvrk_app dvrk_wb_dev > /opt/backups/pre_vXXX_wb_task_$(date +%Y%m%d_%H%M%S).sql
```

Backup не должен заменять git. Используются оба механизма: SQL backup и git tag.

---

## 7. Запреты, которые почти всегда нужны

В task-файлах обязательно задавались запреты. Это критично для безопасности.

Типовой список:

```text
- production не трогать;
- YooKassa API не вызывать на GET-страницах;
- WB API не вызывать на GET-страницах;
- Tilda API не вызывать на GET-страницах;
- не показывать токены;
- не логировать secret key;
- не выводить .env на страницу;
- не ломать ЧЗ-логику;
- не ломать второй закрывающий чек;
- не менять WB raw_json;
- не распределять report-level WB расходы по товарам без отдельного решения;
- не запускать старые проверки без причины.
```

Если задача касается YooKassa, Tilda, WB, логов или настроек, запреты должны быть особенно строгими.

---

## 8. Команды, которые обычно запускал MiMo

Стандартный набор команд в конце задачи:

```bash
cd /opt/dvrk_control_app
source .venv/bin/activate
python -m py_compile app/main.py app/routes/*.py app/core/*.py app/integrations/*.py app/audit/*.py scripts/*.py
systemctl restart dvrk-control-dev
./scripts/smoke_test.sh
python scripts/check_task_vXXX.py
python scripts/check_task_browser_vXXX.py
journalctl -u dvrk-control-dev -n 120 --no-pager
```

Для задач диагностики дополнительно создавались и запускались debug-скрипты:

```bash
python scripts/debug_task_vXXX.py
```

Важно: не запускать все старые проверки v127-v168 без необходимости. Это долго, шумно и может создавать ложные проблемы.

---

## 9. Browser/DOM проверки обязательны для UI

Если задача меняет интерфейс, task-файл должен требовать browser-проверки через Playwright и screenshots.

Примеры screenshots:

```text
/tmp/v168_logs_dashboard.png
/tmp/v168_integrations_top.png
/tmp/v167_stats_saved.png
/tmp/v167_customers_reloaded.png
```

Почему это важно:

- MiMo может написать PASS, но визуально страница будет неудобной;
- пользователь часто оценивает интерфейс по скринам;
- UI-задачи нельзя принимать только по backend-тестам.

---

## 10. Как должен выглядеть report_vXXX.md

В конце задачи MiMo должен создать:

```text
/opt/dvrk_control_app/report_vXXX.md
```

Отчёт должен включать:

- почему задача понадобилась;
- что изменено;
- какие таблицы/файлы созданы;
- какие страницы изменены;
- какие проверки прошли;
- какие screenshots сделаны;
- подтверждение, что старые важные страницы не сломаны;
- подтверждение, что внешние API не вызывались на GET;
- подтверждение, что секреты не показаны;
- текущий git commit;
- текущий git tag.

Пример последних report-файлов:

```text
report_v167.md — UI settings только для /stats и /customers
report_v168.md — новая /logs dashboard и страница /integrations
```

---

## 11. Почему нельзя слепо верить PASS

MiMo может написать, что все проверки прошли, но всё равно нужно проверять:

- скриншоты;
- формулы;
- бизнес-логику;
- противоречия в отчёте;
- не сломаны ли старые сценарии;
- не вызваны ли внешние API;
- не показаны ли secret/token.

Пример из WB: в одном отчёте одновременно было написано, что `AfterWB / 1 = 337 ₽`, и что `AfterWB не изменился и был 354 ₽`. Это противоречие. Даже при PASS задачу нельзя принимать полностью без уточнения.

Правильная оценка в таких случаях:

```text
Техническую часть принимаю, но расчёт/логику пока не принимаю. Нужен следующий контрольный vXXX.
```

---

## 12. Как принимать задачу

После отчёта возможны три варианта.

### Полностью принять

```text
vXXX принимаю.
Текущий tag: ...
```

### Принять частично

```text
vXXX принимаю как базу, но не как финал.
Принято: ...
Не принято: ...
Следующий шаг: vYYY ...
```

### Не принять

```text
vXXX не принимаю.
Причины:
1. ...
2. ...
Нужен корректирующий vYYY.
```

---

## 13. Особенности общения с пользователем

Пользователь предпочитает:

- пошаговую работу;
- короткое объяснение цели;
- готовый bash-блок для создания task-файла;
- одну задачу за раз;
- проверку отчёта и скринов;
- честное указание, что принято, а что нет.

Пользователь не любит:

- хаотичные команды;
- длинные теоретические объяснения вместо task-файла;
- когда ИИ забывает предыдущую логику;
- когда ИИ слепо верит PASS;
- когда интерфейс засоряется лишними блоками;
- когда задача раздувается на весь сайт без необходимости.

---

## 14. Правило одной задачи

Главный принцип:

```text
Одна задача — один task-файл — один запуск MiMo — один отчёт.
```

Если задача большая, её нужно делить.

Пример:

```text
v168 — сделать компактную /logs и создать /integrations
v169 — доработать /integrations до полноценной панели
v170 — улучшить операторский workflow /orders
```

Не нужно смешивать всё в один огромный task.

---

## 15. Почему GET-страницы не должны вызывать внешние API

В проекте есть интеграции:

- YooKassa;
- Tilda;
- Wildberries;
- потенциально email/Telegram в будущем.

Открытие страницы должно быть безопасным. GET-запрос должен только читать данные из БД и показывать HTML.

Критичные запреты:

```text
GET /orders не должен отправлять чек.
GET /chz не должен вызывать YooKassa.
GET /wb не должен вызывать WB API.
GET /integrations не должен проверять YooKassa/Tilda автоматически.
GET /logs не должен вызывать внешние API.
```

Проверки внешних подключений допустимы только по явному POST-действию пользователя, например кнопка "Проверить подключение".

---

## 16. Как работали с git

После успешной задачи MiMo делал commit и tag.

Примеры последних tags:

```text
v1.63-wb-adjustments-normalization
v1.64-stats-customers-ui-settings
v1.65-logs-integrations-settings
```

В отчёте обязательно указывать:

```text
Commit: ...
Tag: ...
```

Если задача не принята, следующий task должен исправлять конкретные проблемы, а не переписывать всё с нуля.

## 18. Главный вывод для другого ИИ

Не надо начинать улучшать всё сразу.

Нужно продолжать так:

```text
понять текущую задачу;
создать vXXX.md;
дать пользователю bash-блок;
попросить MiMo выполнить;
получить report_vXXX.md;
проверить отчёт и скрины;
принять или создать следующий корректирующий vXXX.
```

Этот процесс уже доказал свою эффективность для DVRK Control. Его нужно сохранить.
