# 10 — Логи, аудит и backup-система

Этот файл описывает раздел `/logs`, систему аудита, логи, backup, Windows pull backup и правила безопасной работы с диагностикой в проекте DVRK Control.

Файл предназначен для другого ИИ-агента. Его нужно читать перед любыми изменениями в `/logs`, audit events, системных логах, backup, скачивании архивов, helper-блоках, интеграционных логах и диагностике.

---

## 1. Роль страницы `/logs`

`/logs` — это техническая панель контроля системы.

Она нужна не как обычная пользовательская страница, а как рабочий центр для:

```text
просмотра событий;
просмотра ошибок;
контроля webhook Tilda;
контроля YooKassa;
контроля WB API;
контроля backup;
скачивания backup;
быстрой диагностики;
понимания, что происходило в системе.
```

Страница должна помогать оператору и разработчику быстро понять состояние системы.

---

## 2. Что должно быть в `/logs`

Основные блоки:

```text
верхние summary cards;
таблица событий / audit;
фильтры;
page size 50 / 100 / 200;
быстрые действия;
состояние backup;
скачивание backup;
системный помощник;
информация о последних ошибках.
```

После v168 `/logs` был переделан в компактный dashboard.

Это направление принято как хорошая база.

---

## 3. Что было до v168

До v168 страница `/logs` была слишком длинной и неудобной.

Проблема:

```text
много информации вертикально;
непонятная структура;
сложно быстро увидеть главное;
backup и события были смешаны;
страница выглядела как длинное техническое полотно.
```

Пользователь попросил:

```text
навести порядок;
сделать компактно;
сделать профессионально;
не терять важную диагностику.
```

---

## 4. Что сделано в v168

В v168 `/logs` был переделан в dashboard:

```text
верхние карточки статусов;
левая часть — события и фильтры;
правая часть — backup, быстрые действия, системный помощник;
таблица событий стала компактнее;
добавлены page size 50 / 100 / 200;
сохранена диагностическая функция страницы.
```

Отчёт v168 был принят как рабочая база по `/logs`.

---

## 5. Важное правило по `/logs`

`/logs` не должен вызывать внешние API при обычном открытии страницы.

GET `/logs` может:

```text
читать локальную БД;
читать локальные файлы логов, если это безопасно;
читать список локальных backup;
показывать последние локально сохранённые статусы;
показывать audit history.
```

GET `/logs` не должен:

```text
вызывать YooKassa API;
вызывать Tilda API;
вызывать WB API;
создавать чек;
запускать синхронизацию;
перезаписывать настройки;
создавать backup без отдельного POST/action.
```

---

## 6. Аудит

Аудит — это технический журнал действий и событий.

Он нужен для:

```text
разбора ошибок;
понимания действий оператора;
отслеживания webhook;
контроля отправки чеков;
отслеживания системных событий;
проверки работы ИИ-агента;
контроля backup.
```

Аудит не должен быть красивой CRM-историей для клиента. Это внутренний технический журнал.

---

## 7. Какие события писать в аудит

Важно писать:

```text
получен webhook Tilda;
обнаружен duplicate webhook;
создан заказ;
изменён заказ;
введён код ЧЗ;
изменён код ЧЗ;
открыт preview закрывающего чека;
открыт confirm отправки чека;
оператор подтвердил отправку;
backend заблокировал отправку;
чек отправлен в YooKassa;
YooKassa вернула pending;
YooKassa вернула succeeded;
YooKassa вернула failed;
ошибка YooKassa API;
попытка повторной отправки заблокирована;
WB синхронизация запущена;
WB синхронизация завершена;
WB API ошибка;
настройки интеграций изменены;
backup создан;
backup скачан;
backup ошибка;
системная ошибка.
```

---

## 8. Что нельзя писать в аудит

Нельзя писать:

```text
YooKassa secret key;
WB token;
Tilda webhook secret;
полный Authorization header;
cookies;
.env;
пароли;
полные sensitive payload без очистки;
персональные данные сверх необходимого.
```

Если нужно сохранить payload для диагностики, он должен быть безопасно очищен.

---

## 9. Логи ошибок

Ошибки должны быть понятны.

Для оператора:

```text
короткий статус;
что произошло;
где произошло;
что делать дальше.
```

Для разработчика:

```text
trace_id / event_id;
endpoint;
exception type;
безопасный message;
timestamp;
связанный order_id / report_id, если есть.
```

Не нужно выводить весь traceback в пользовательский интерфейс, если он содержит чувствительные данные.

---

## 10. Системные логи

Для диагностики на сервере может использоваться:

```bash
journalctl -u dvrk-control-dev -n 200 --no-pager
```

Или аналогичная команда для текущего systemd-сервиса.

Важно:

```text
логи сервиса не должны содержать токены;
не выводить .env;
не печатать Authorization headers.
```

---

## 11. Backup: назначение

Backup нужен, чтобы восстановить проект после ошибки ИИ, сбоя сервера или неправильной миграции.

Backup должен покрывать:

```text
код проекта;
важные конфиги без публичного раскрытия;
базу данных / dump;
важные локальные данные;
миграции;
скрипты;
отчёты, если они нужны.
```

Точная структура зависит от текущих скриптов проекта.

---

## 12. Backup-папка

В проекте использовалась backup-директория:

```text
/opt/dvrk_backups/daily/
```

Пример файла backup:

```text
dvrk_full_backup_2026-07-02_17-18-25.tar.gz
```

---

## 13. Backup на странице `/logs`

На `/logs` должен быть блок backup.

Он должен показывать:

```text
последний backup;
дату и время;
размер;
статус;
кнопку скачать;
ошибку, если backup failed;
краткую инструкцию.
```

Не нужно перегружать блок длинным текстом.

---

## 14. Скачивание backup

Скачивание backup должно быть безопасным.

Правила:

```text
разрешать скачивать только файлы из разрешённой backup-директории;
запрещать path traversal;
не давать скачать произвольные файлы сервера;
не давать скачать .env напрямую;
не показывать секреты в списке файлов.
```

Плохо:

```text
/download?path=/etc/passwd
```

Правильно:

```text
/download-backup/{backup_id}
```

где `backup_id` мапится на файл внутри разрешённой директории.

## 15. Windows pull backup

Была настроена схема скачивания backup на Windows.

Локальная папка:

```text
C:\DVRK_BACKUPS
```

SSH-ключ:

```text
C:\Users\mrdob\.ssh\dvrk_backup_ed25519
```

Первый успешно скачанный backup:

```text
dvrk_full_backup_2026-07-02_17-18-25.tar.gz
```

Была проблема с кириллицей в PowerShell логах, но сам backup скачивался.

---

## 16. Что важно про Windows backup

Не нужно ломать рабочую схему.

Если другой ИИ будет менять backup:

```text
сначала проверить текущий script;
не менять путь C:\DVRK_BACKUPS без причины;
не менять ключ без причины;
не ломать cron/systemd backup;
проверить скачивание фактического файла;
не считать проблему кодировки PowerShell критичной, если файл скачивается.
```

---

## 17. Backup перед задачами MiMo

Перед рискованными изменениями MiMo должен делать backup.

Особенно перед задачами:

```text
миграции БД;
изменение схемы заказов;
изменение YooKassa;
изменение Tilda webhook;
изменение WB нормализации;
изменение backup-системы;
удаление данных;
массовые пересчёты.
```

В task-файле нужно явно писать:

```text
Сначала сделай backup.
```

---

## 18. Git и backup

Git не заменяет backup.

Git защищает код.

Backup защищает:

```text
данные;
БД;
локальные файлы;
отчёты;
runtime-состояние;
настройки.
```

Перед изменениями желательно иметь:

```bash
git status
git tag
backup archive
```

---

## 19. Отчёты MiMo

После каждой задачи MiMo должен создавать report-файл.

Пример:

```text
report_v168.md
```

Report нужен для проверки.

В отчёте должны быть:

```text
что изменено;
какие файлы изменены;
какие проверки прошли;
какие browser screenshots;
какие ошибки были;
что не трогалось;
git commit;
git tag.
```

---

## 20. Проверка отчётов

Нельзя слепо принимать `PASS`.

Нужно сверять:

```text
что задача реально выполнена;
что не сломались соседние страницы;
что GET не вызывает внешние API;
что секреты не попали в HTML/logs;
что screenshots соответствуют описанию;
что git status чистый;
что tag создан.
```

Пользователь часто просит: "проверь всё внимательно".

---

## 21. Системный помощник на `/logs`

После v168 на `/logs` есть блок системного помощника.

Смысл:

```text
дать быстрые подсказки по диагностике;
показать команды;
объяснить, что делать при ошибке;
не засорять основную таблицу.
```

Помощник должен быть компактным.

---

## 22. Быстрые действия

На `/logs` могут быть быстрые действия:

```text
открыть /integrations;
открыть /orders;
открыть /chz;
открыть /wb/logs;
скачать последний backup;
посмотреть последние ошибки;
обновить страницу.
```

Опасные действия не должны запускаться без confirm / POST.

---

## 23. Связь с `/integrations`

`/integrations` хранит настройки YooKassa/Tilda.

`/logs` может показывать историю изменений этих настроек.

Важно:

```text
показывать masked values;
не показывать старый/новый secret полностью;
не выводить token в audit.
```

---

## 24. Связь с YooKassa

В логах можно показывать:

```text
receipt_id;
order_id;
status;
ошибку YooKassa;
timestamp;
safe request summary;
safe response summary.
```

Нельзя показывать:

```text
secret key;
Authorization header;
полный sensitive payload.
```

---

## 25. Связь с Tilda

В логах можно показывать:

```text
получен webhook;
Tilda order id;
payment[systranid];
количество товаров;
статус обработки;
ошибку parser.
```

Нельзя показывать всё подряд, если payload содержит лишние персональные данные или секреты.

---

## 26. Связь с WB

WB имеет отдельную страницу:

```text
/wb/logs
```

Но `/logs` может показывать общие системные события WB:

```text
WB sync started;
WB sync completed;
WB sync failed;
rate limit;
token missing;
API error.
```

GET `/logs` не должен запускать WB sync.

---

## 27. Rate limit и внешние API

Логи должны помогать увидеть, если система спамит API.

Особенно важно для WB:

```text
не запускать WB API часто;
ручная синхронизация;
cooldown;
GET-страницы не вызывают WB API.
```

Для YooKassa:

```text
не проверять API на каждом page load;
polling только для pending/sent receipt;
не делать повторную отправку.
```

---

## 28. UI `/logs`

UI должен быть:

```text
компактный;
тёмный;
профессиональный;
без длинного полотна;
с понятными фильтрами;
с page size;
с правой панелью backup/helper.
```

Не возвращать старый вид.

---

## 29. Фильтры логов

Полезные фильтры:

```text
тип события;
уровень: info / warning / error;
источник: tilda / yookassa / wb / backup / system;
период;
order_id;
receipt_id;
report_id;
текстовый поиск.
```

Фильтры не должны ломать производительность.

---

## 30. Page size

Для таблицы событий использовать:

```text
50
100
200
```

Не грузить тысячи событий сразу без необходимости.

---

## 31. Производительность

`/logs` может стать тяжёлой страницей.

Нужно:

```text
ограничивать количество событий;
использовать пагинацию;
не читать огромные файлы целиком;
не делать внешние API вызовы;
не блокировать UI долгими операциями.
```

---

## 32. Безопасность `/logs`

Главные запреты:

```text
не показывать .env;
не показывать tokens;
не показывать secrets;
не показывать Authorization;
не давать скачать произвольный файл;
не запускать shell-команды из UI без строгого контроля;
не делать destructive actions без confirm.
```

---

## 33. Проверки после изменений `/logs`

Проверить:

```text
/logs открывается;
таблица событий отображается;
фильтры работают;
page size работает;
backup block отображается;
скачивание backup безопасно;
нет JS errors;
секреты не видны;
GET /logs не вызывает внешние API;
/orders открывается;
/chz открывается;
/integrations открывается;
/wb открывается.
```

Если задача затрагивает `/logs` или backup, report должен содержать:

```text

как проверена защита от path traversal;
как проверено отсутствие secrets;
какие GET-запросы не вызвали внешние API;
какие screenshots сохранены;
какие checks прошли;
git commit и tag.
```

## 36. Нельзя делать без отдельного решения

Нельзя:

```text
удалять audit history;
переносить backup-папку;
отключать backup;
делать публичный доступ к backup;
добавлять shell executor в UI;
показывать .env;
логировать токены;
запускать внешние API на GET;
делать массовую очистку логов без подтверждения.
```
