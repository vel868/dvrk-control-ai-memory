# 16 — Security and DO NOT BREAK: критичные запреты проекта

Этот файл фиксирует самые важные правила безопасности и стабильности DVRK Control. Его нужно читать перед любой новой задачей MiMo или другого ИИ-агента.

Главная цель файла — не дать новому агенту случайно сломать рабочую логику Tilda / YooKassa / Честный Знак / WB / backup / UI.

---

## 1. Главный принцип

DVRK Control уже содержит стабильные рабочие части.

Перед любым изменением нужно думать не только:

```text
как добавить новую функцию
```

но и:

```text
что можно случайно сломать;
какие внешние API можно случайно вызвать;
какие реальные чеки можно случайно отправить;
какие токены можно случайно раскрыть;
какие данные можно испортить.
```

---

## 2. Самое опасное в проекте

Критичные зоны:

```text
YooKassa closing receipts;
Tilda webhook parser;
Честный Знак через SKU ZH;
WB API token и финансовая нормализация;
backup download;
secrets в /integrations;
raw_json WB;
расчёт скидок;
report-level расходы WB;
GET-страницы и внешние API.
```

Любая задача в этих зонах должна быть максимально аккуратной.

---

## 3. Нельзя делать первый чек в DVRK Control

Стабильная схема:

```text
Tilda принимает оплату.
Tilda / YooKassa формируют первый чек предоплаты.
DVRK Control формирует только второй закрывающий чек.
```

Запрещено:

```text
добавлять первый чек в DVRK Control;
автоматически отправлять чек сразу после webhook;
делать чек оплаты вместо закрывающего чека;
смешивать payment и closing receipt.
```

---

## 4. Второй закрывающий чек: запреты

Нельзя:

```text
отправлять второй чек при full_payment;
отправлять чек при unknown payment_mode;
отправлять чек без кода ЧЗ для ZH-товара;
отправлять чек, если сумма не сходится с оплатой;
отправлять чек повторно, если receipt succeeded;
отправлять чек повторно, если receipt pending;
отправлять чек на GET-странице;
обходить backend-проверки только потому, что UI disabled.
```

Backend обязан повторно проверять безопасность перед отправкой.

---

## 5. Честный Знак: запреты

Главное правило:

```text
ZH в SKU = нужен Честный Знак.
```

Нельзя:

```text
менять ZH на XH;
требовать ЧЗ для товара без ZH;
не требовать ЧЗ для товара с ZH;
добавлять mark_code_info доставке;
добавлять mark_code_info обычному товару;
кодировать код ЧЗ в Base64;
передавать ЧЗ в YooKassa не через mark_code_info.gs_1m.
```

---

## 6. YooKassa secrets

Нельзя показывать:

```text
YooKassa Secret Key;
Authorization header;
.env;
полный API payload с чувствительными данными;
полный response, если там есть чувствительные данные.
```

Где нельзя показывать:

```text
HTML;
API response;
audit;
logs;
report_vXXX.md;
screenshots;
console output;
browser localStorage.
```

В UI только masked.

---

## 7. Tilda secrets и webhook

Нельзя:

```text
показывать Tilda webhook secret/token полностью;
логировать его;
затирать старый secret пустым полем;
включать application/json без переписывания parser;
отключать передачу товаров массивом;
отключать "отправлять только после оплаты";
менять payment[systranid] как YooKassa payment_id без отдельной задачи.
```

Стабильные настройки Tilda:

```text
API METHOD: Off
Товары массивом: Включено
Отправлять только после оплаты: Включено
application/json: Выключено
Cookies: Выключено
externalid: Выключено
```

---

## 8. WB token

WB token — secret.

Нельзя:

```text
показывать WB token полностью;
логировать WB token;
писать WB token в audit;
отдавать WB token через API;
сохранять WB token в UI settings;
вставлять WB token в report_vXXX.md.
```

Показывать только masked.

---

## 9. GET-страницы не вызывают внешние API

Это одно из главных правил проекта.

GET-страницы не должны вызывать:

```text
YooKassa API;
Tilda API;
WB API;
отправку чеков;
синхронизацию WB;
проверку подключений;
создание backup, если это тяжёлая операция.
```

GET может читать только локальную БД и локальные безопасные файлы.

Активные действия — только через POST/action.

---

## 10. WB API: запреты

Нельзя:

```text
вызывать WB API на GET /wb*;
запускать синхронизацию автоматически при открытии страницы;
обходить cooldown;
логировать token;
удалять raw_json;
считать deliveryAmount рублями;
считать returnAmount рублями;
считать все quantity продажами;
повторно вычитать эквайринг после forPay;
называть forPay чистой прибылью.
```

WB sync должен быть ручным и контролируемым.

---

## 11. WB raw_json

`raw_json` — критичный источник.

Нельзя:

```text
удалять raw_json;
перезаписывать raw_json очищенными данными;
терять исходные поля WB;
мигрировать raw_json без backup;
считать, что все важные поля уже нормализованы.
```

Если WB добавил новое поле, его сначала нужно найти в raw_json и диагностировать.

---

## 12. WB report-level расходы

Особенно важно:

```text
deduction;
paidStorage;
paidAcceptance;
penalty;
bankPaymentSum.
```

Нельзя:

```text
автоматически распределять report-level расходы по товарам без отдельного режима;
молча размазывать deduction по всем товарам;
скрывать крупные удержания;
делать вид, что unit economics полностью точная, если report-level расходы не распределены.
```

Нужно показывать честное предупреждение.

---

## 13. Финансовые формулы: запреты

Нельзя ломать модель скидок Tilda:

```text
goods_gross_sum
goods_discount_sum
goods_net_sum
delivery_sum
order_total_paid
```

Главная формула:

```text
goods_net_sum + delivery_sum = order_total_paid
```

Нельзя:

```text
распределять скидку на доставку;
смешивать доставку с товарной выручкой;
строить закрывающий чек по суммам до скидки;
отправлять чек, если итог не сходится.
```

---

## 14. WB unit economics: запреты

Нельзя:

```text
считать forPay финальной выплатой за товар;
считать bankPaymentSum точной выплатой за товар;
делить bankPaymentSum на количество и называть это точной unit economics;
игнорировать logistics после forPay;
игнорировать deduction;
показывать расходы без минуса;
скрывать correction_to_forpay;
смешивать остаток после WB и чистую прибыль.
```

Правильно:

```text
forPay = WB начислил по товарной строке;
Остаток после WB = forPay − расходы WB после forPay;
Чистая прибыль = остаток после WB − расходы продавца.
```

---

## 15. Backup: запреты

Нельзя:

```text
отключать backup;
удалять backup без отдельного решения;
менять backup directory без необходимости;
делать backup публично доступным;
давать скачать произвольный файл сервера;
скачивать /etc/passwd, .env и другие системные файлы через backup endpoint;
запускать destructive-действия без backup.
```

Backup directory:

```text
/opt/dvrk_backups/daily/
```

Windows pull backup:

```text
C:\DVRK_BACKUPS
C:\Users\mrdob\.ssh\dvrk_backup_ed25519
```

---

## 16. Download backup security

Endpoint скачивания backup должен:

```text
разрешать только файлы из backup-директории;
запрещать path traversal;
не принимать произвольный абсолютный путь;
не отдавать .env;
не отдавать системные файлы.
```

Плохо:

```text
/download?path=/etc/passwd
```

Лучше:

```text
/download-backup/{backup_id}
```

---

## 17. UI settings: запреты

В v167 сохранение UI-настроек разрешено только для:

```text
/stats
/customers
```

Нельзя:

```text
подключать user_ui_settings ко всему сайту без отдельной задачи;
сохранять tokens/secrets/passwords в UI settings;
сохранять персональные данные покупателей как настройки;
делать глобальный сброс всего сайта;
сбрасывать данные заказов через кнопку "Сбросить настройки страницы".
```

Кнопка сброса должна сбрасывать только UI-состояние страницы.

## 18. CRM и персональные данные

Нельзя:

```text
определять пол покупателя по имени;
склеивать покупателей только по имени;
делать маркетинговые рассылки без согласия Marketing;
автоматически отправлять сообщения клиентам;
логировать лишние персональные данные;
показывать персональные данные в debug без необходимости.
```

Мужское / Женское — это секции товаров, не пол покупателя.

---

## 19. Интеграции `/integrations`

Нельзя:

```text
показывать secrets полностью;
затирать secret пустым input;
вызывать внешние API на GET;
отправлять тестовый чек без confirm;
логировать старый/новый secret в audit;
ломать /tilda, если старая страница ещё используется.
```

Всё active — через POST/action.

---

## 20. Logs и audit

Нельзя в logs/audit писать:

```text
YooKassa secret key;
WB token;
Tilda secret;
Authorization headers;
cookies;
.env;
полные sensitive payload;
пароли.
```

Audit должен быть полезным, но безопасным.

## 21. UI: что нельзя ломать

Пользователь предпочитает:

```text
тёмную тему;
компактный интерфейс;
русский язык;
центрированные страницы;
тонкие тёмные скроллбары;
tooltip вместо лишних видимых блоков.
```

Нельзя:

```text
возвращать длинные полотна;
засорять страницы постоянными пояснениями;
ломать таблицы CSS grid на table/tr/td;
убирать причины disabled-кнопок;
делать опасные кнопки без confirm.
```

## 22. Logs UI после v168

`/logs` после v168 принят как хорошая база.

Нельзя:

```text
возвращать старое длинное полотно;
смешивать backup и события хаотично;
убирать компактный dashboard;
запускать внешние API при открытии /logs;
показывать secrets.
```

## 23. WB signs in UI

Расходы WB должны отображаться со знаком минус:

```text
− Комиссия WB / 1
− Эквайринг / 1
− Логистика WB / 1
− Обратная логистика WB / 1
− Расходы WB после forPay / 1
```

Доплаты — с плюсом:

```text
+ Доплаты
```

Корректировка до forPay может быть:

```text
+/- Корректировка до forPay / 1
```

---

## 24. MiMo workflow: запреты

Пользователь работает пошагово.

Нельзя:

```text
давать сразу огромную пачку задач;
просить пользователя вручную править код, если можно дать task-файл;
прыгать далеко вперёд;
слепо верить report PASS;
создавать задачу без проверок;
не требовать screenshots для UI;
не требовать git tag.
```

Правильный формат:

```text
один vXXX.md;
одна задача;
один запуск MiMo;
один report_vXXX.md;
проверка;
следующая задача.
```

---

## 25. Нельзя принимать PASS без проверки

MiMo может написать:

```text
PASS
```

Но это не означает, что всё реально правильно.

Нужно сверять:

```text
screenshots;
browser checks;
логические сценарии;
GET no external API;
secret masking;
формулы;
суммы;
git status;
git tag.
```

---

## 26. Git safety

Перед изменениями:

```bash
git status
```

После изменений:

```bash
git status
git diff
```

Для принятой задачи нужен commit и tag.

Нельзя оставлять рабочую директорию в грязном состоянии без объяснения.

---

## 27. Systemd safety

Сервис:

```text
dvrk-control-dev
```

Нельзя:

```text
перезапускать production-сервис, если задача про dev;
менять service unit без backup;
оставлять сервис в failed;
не проверять journalctl после restart.
```

---

## 28. Database safety

Перед миграциями:

```text
backup;
описать миграцию;
проверить существующие данные;
не удалять столбцы без отдельного решения;
не делать destructive migration без rollback-плана.
```

Нельзя:

```text
очищать реальные заказы;
удалять WB raw rows;
удалять audit history;
перезаписывать финансовые данные без backup.
```

---

## 29. Чего нельзя делать без отдельного решения пользователя

Нельзя без отдельного явного решения:

```text
включать уведомления email/Telegram;
переносить WB token в /integrations;
распределять WB report-level расходы;
менять схему ЧЗ с ZH;
менять стабильные Tilda настройки;
переводить Tilda webhook на JSON;
удалять складской модуль;
возвращать склад Tilda в UI как основной раздел;
делать глобальную авторизацию;
включать live YooKassa mode.
```

---

## 30. Уведомления пока не делать

Пользователь отдельно решил:

```text
уведомления email / Telegram пока не делать.
```

Не нужно создавать задачу на уведомления без нового запроса.

---

## 31. Проверки для любой задачи

Минимально после любой задачи:

```text
страница, которую меняли, открывается;
соседние страницы открываются;
нет JS errors;
нет secrets в HTML/API/logs;
GET не вызывает внешние API;
py_compile проходит;
smoke проходит;
git status понятен.
```

---

## 32. Проверки для YooKassa / ЧЗ

Проверить:

```text
ZH + full_prepayment + нет кода → нельзя отправить;
ZH + код → можно готовить чек;
non-ZH + full_prepayment → чек без ЧЗ;
full_payment → чек не нужен;
unknown → проверить признак расчёта;
суммы сходятся;
повторная отправка blocked.
```

---

## 33. Проверки для Tilda

Проверить:

```text
webhook parser работает;
товары массивом читаются;
payment[systranid] сохраняется;
SKU читается;
ZH определяется;
скидка считается;
доставка отдельно;
duplicate webhook не создаёт дубль.
```

---

## 34. Проверки для WB

Проверить:

```text
GET /wb* не вызывает API;
token masked;
rrdId не ломается;
raw_json сохраняется;
forPay считается;
bankPaymentSum не путается с товарной выплатой;
deliveryAmount/returnAmount не деньги;
massagor и lopata0126 контрольные товары;
расходы со знаком минус.
```

---

## 35. Проверки для UI

Проверить:

```text
Playwright/browser screenshots;
нет JS errors;
таблицы не развалились;
disabled-кнопки имеют причину;
опасные действия имеют confirm;
tooltip работает;
тёмная тема сохранена;
страницы центрированы.
```

---

## 36. Что писать в report_vXXX.md

В отчёте обязательно:

```text
что изменено;
какие файлы изменены;
что не трогалось;
какие проверки прошли;
какие сценарии проверены;
где screenshots;
секреты не видны;
GET no external API;
git commit;
git tag;
известные ограничения.
```

---

## 37. Красные флаги в report

Если в report есть такие фразы, нужно перепроверить:

```text
"скорее всего";
"не проверялось";
"не удалось";
"временно";
"позже исправить";
"PASS", но без screenshots;
"GET проверку не делал";
"секреты не проверял";
"формулы примерно сходятся";
```

Это не обязательно ошибка, но требует внимания.

---

## 38. Главное резюме

Кратко:

```text
Не отправлять реальные внешние действия на GET.
Не раскрывать secrets.
Не ломать Tilda → YooKassa → второй чек.
ZH в SKU — критичное правило ЧЗ.
WB API только ручной sync/action.
WB raw_json сохранять.
forPay не чистая прибыль.
Report-level WB расходы не распределять без отдельного режима.
Backup перед рискованными задачами.
UI проверять browser screenshots.
PASS MiMo не принимать без проверки.
Уведомления пока не делать.
```
